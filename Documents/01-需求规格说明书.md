## 1. 产品定位：AI 面试教练

OfferPilot 定位为**求职面试全流程 AI 助手**，覆盖从简历诊断、模拟面试到录音复盘的核心环节。

```
核心流程：

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  简历诊断  │ →  │  模拟面试  │ →  │  面试复盘  │ →  │  知识检索  │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↓                ↓                ↓                ↓
  解析 PDF 简历     AI 扮演面试官     录音转写 +        检索面试题库
  多维度评估       4 种面试模式      六维深度分析      优秀答案对标
  优化建议         追问机制          改进建议          联网搜索兜底
```

## 2. 四大功能模块（功能性需求）

### 功能 1：简历智能诊断

用户上传 PDF 简历，Agent 解析内容，从多个维度评估并给出优化建议。

**Agent 能做的事：**
- 解析简历结构（教育背景、项目经历、技能清单、实习经历）
- 对比目标岗位的 JD（职位描述），计算关键词匹配度
- 检查 STAR 法则的运用（项目描述是否有情境-任务-行动-结果）
- 识别常见问题（技能罗列没有层次、项目描述只写技术栈没写成果、缺少量化数据）
- 生成优化后的简历建议（不是替用户写，而是告诉用户哪里该改、怎么改）

**技术含量：** PDF 解析（Apache PDFBox）、结构化输出（JSON Schema 约束简历评估报告）、RAG 检索优秀简历模板和 HR 筛选标准。

**企业技能展示：** 文件上传/解析、异步任务（PDF 解析耗时，需要异步处理 + 进度通知）、报告生成。

### 功能 2：AI 模拟面试（核心功能）

Agent 扮演面试官，和用户进行多轮对话模拟。支持不同面试模式。

**面试模式：**
- **技术深挖模式（TECH_DEEP）**：Agent 根据用户简历中的项目经历，深挖技术细节
- **行为面试模式（BEHAVIOR）**：经典的 STAR 问题，评估软素质
- **系统设计模式（SYSTEM_DESIGN）**：给一个需求，让用户设计架构
- **压力面试模式（PRESSURE）**：Agent 会追问、质疑、施加压力

**技术含量：** 多轮对话状态管理、角色扮演的 System Prompt 工程、对话历史上下文压缩（CompactionConfig, 阈值 8000 tokens）、SSE 流式输出。

**企业技能展示：** SSE 实时通信、会话状态持久化（Redis AgentStateStore）、上下文管理、HITL 人工确认机制（高危工具调用需用户确认）。

### 功能 3：面试录音/文字分析

用户上传面试录音或粘贴面试文字记录，Agent 做深度分析。

**分析维度（6 个）：**

| 维度 | 分析内容 | 评分方式 |
|:---|:---|:---|
| 技术深度 | 回答是否触及原理、是否有深度对比 | RAG 检索标准答案对标 |
| 表达逻辑 | 是否有清晰的结构（总分总/STAR）、是否有废话 | LLM 评估 |
| 知识盲区 | 哪些问题完全答不上来或答错了 | RAG 检索正确答案 |
| 时间分配 | 每个问题的回答时长是否合理 | 时间戳分析 |
| 自信度 | 语气词频率、犹豫点检测 | 文本特征分析 |
| 亮点时刻 | 回答中特别出色的部分 | RAG 检索优秀答案对标 |

**技术含量：** 音频转文字（DashScope Paraformer ASR）、多维度评估的结构化输出、RAG 精准检索对标答案。

**企业技能展示：** 第三方服务集成（ASR）、复杂报告生成、数据可视化。

### 功能 4：知识检索与智能问答

Agent 通过多路检索 + 联网兜底回答用户的面试相关知识问题。

**Agent 能做的事：**
- 检索面试题库（search_questions）、优秀答案（search_answers）、学习资源（search_resources）
- 智能路由检索（smart_search）：自动识别查询意图，路由到对应知识库子库
- 列出可访问的知识库及文档统计（list_knowledge_bases）
- 知识库无结果或相关性不足时，自动触发 MCP 联网搜索（search）兜底

**技术含量：** Milvus 向量检索 + 标量过滤混合检索、MCP 协议集成百度搜索、多级召回 → RRF 融合 → Reranker 精排。

## 3. 完整对话流程示例（4 个场景 / 用例描述）

### 示例 1：面试录音分析（多 Agent 并行）

```
用户: [上传 interview_20260708.mp3] 帮我分析一下刚才的面试录音

主Agent 推理: 音频分析 → 先转文字，再并行评估

  [第一步：主 Agent 调用 transcribe_audio]
  工具返回: 带时间戳的文字记录（5 个问答对）

  [第二步：对每个问答对，并行 spawn 两个子 Agent]

  [tech_evaluator 分析第 1 题：HashMap 底层]
  调用 search_answers(question="HashMap底层原理")
    → KnowledgeBaseService.searchWithTenant()
    → 公共库命中 + 用户私有库（如有）命中
    → 返回 Top-5 优秀答案（相似度 0.89~0.94）
  调用 analyze_answer(question="...", answer="...", reference="...")
  结果: 技术深度 75分，覆盖了数组+链表，遗漏了红黑树转换条件和尾插法

  [expression_evaluator 分析第 1 题]
  结果: 表达逻辑 60分，没有先给结论再展开，"嗯"出现 4 次

  [所有题目分析完毕]
  主Agent 整合:
  "你的面试录音分析结果：
  总体评分：72/100（较上次 +5 分）
  Q1 HashMap（75分）：底层结构讲清楚了，但 JDK 1.8 的优化没提到
  Q2 Spring 事务（80分）：传播机制答得不错
  Q3 Redis 缓存（65分）：穿透和击穿的区别说对了，但解决方案不完整
  薄弱点：Redis 相关问题的得分普遍偏低，建议系统复习"
```

### 示例 2：模拟面试（多轮对话）

```
用户: 我想模拟面试，目标是字节跳动 Java 后端

Agent: 调用 generate_next_question → 获取出题指导
Agent: "来，先聊个基础的。看你简历上写了熟悉 HashMap，能讲讲底层实现吗？"
用户: "HashMap 底层是数组加链表，JDK 1.8 之后加了红黑树..."

Agent: 调用 analyze_answer → 获取评估指导 → 分析回答质量
Agent: "红黑树的转换条件能具体说说吗？为什么阈值选的是 8？"

... (继续 5-8 题) ...

Agent: "面试结束，给你做个总结：技术广度 78，深度 65。建议重点补红黑树数学原理和 Redis 高级特性。"
```

### 示例 3：知识检索与联网兜底

```
用户: 字节跳动 Java 后端面试都考什么？

Agent: 步骤 1 — 意图识别：company（公司情报查询）
Agent: 步骤 2 — 调用 smart_search → 知识库检索
  → 结果：知识库中有匹配信息
Agent: 步骤 4 — 整合输出："根据知识库信息，字节跳动 Java 后端高频考点包括：
  算法（LRU缓存、二叉树）、Java（HashMap底层、JVM）、Redis..."

---

用户: 2025 年最新的大模型面试趋势是什么？

Agent: 步骤 2 — 调用 smart_search → 知识库无结果（total==0）
Agent: 步骤 3 — 触发 search（MCP 联网搜索）→ 百度搜索最新信息
Agent: 步骤 4 — 整合输出："根据最新信息，2025年大模型面试趋势...（引用联网搜索结果）"
```

## 4. 企业级技能覆盖清单（需求追溯矩阵）

| 技能类别 | 在 OfferPilot 中的体现 |
|:---|:---|
| 用户认证 | Spring Security + JWT 无状态认证、BCrypt 密码加密、Token 过期管理、JWT jti 黑名单机制 |
| 角色权限 | USER/ADMIN 双角色（UserRole 枚举）、接口权限矩阵 |
| 数据隔离 | 基于 userId 的数据隔离、多租户知识库（PUBLIC/PRIVATE 枚举） |
| 分层架构 | Controller → Service → Repository 三层分离，Controller 不直接访问 Repository |
| 审计基类 | BaseEntity（@MappedSuperclass + @EntityListeners），自动填充 createdAt/updatedAt |
| 枚举规范 | 7 个枚举类（UserRole/Visibility/DocumentStatus/InterviewMode/MemoryCategory/SessionStatus/ProviderPreset） |
| DTO 校验 | @Valid + JSR-303 注解（@NotBlank/@Size/@Email），GlobalExceptionHandler 统一返回字段级错误 |
| 统一响应 | ApiResponse\<T\> 统一包装 {code, message, data}，所有接口返回格式一致 |
| 异常体系 | BusinessException 基类 + 7 个业务子类，GlobalExceptionHandler 按 httpStatus 动态响应 |
| 本地缓存 | Caffeine 有界缓存（Agent 池，maximumSize=500 + expireAfterAccess=30min），防 OOM |
| 向量数据库 | Milvus 2.4.6 Standalone、动态 Collection 创建、IVF_FLAT 索引、混合检索（向量+标量过滤） |
| Embedding | DashScope text-embedding-v3（1024 维）、独立 API Key 配置、批量处理 |
| RAG 管道 | 文档解析→分块→Embedding→入库→检索 全链路，4 种分块策略 |
| 动态知识库 | 前端上传→异步入库→进度追踪→检索测试→重建索引→多租户隔离 |
| Redis 缓存 | 会话短期记忆（Redis StateStore）、检索结果缓存（SearchResultCacheService）、接口限流（RateLimitService） |
| 文件处理 | PDF/DOCX/MD/TXT 多格式解析、音频上传转写 |
| 异步处理 | @Async + 线程池、异步入库管道、进度轮询 |
| SSE 流式推送 | streamEvents() + SseEmitter 实时推送，支持思考可见性、HITL 确认 |
| 数据库设计 | 17 张表（含 Token 黑名单、检索日志、模型配置、聊天会话/消息），实体继承 BaseEntity |
| 单 Agent 全能助手 | HarnessAgent 承载全部能力，SysPrompt 中定义意图识别→工具调用流程 |
| 多 Provider 切换 | 用户私有模型 > 用户默认 > 全局默认 > application.yml 四级优先级，支持 DashScope/DeepSeek 等多种 Provider |
| MCP 协议 | McpClientBuilder 直连 OpenWebSearch MCP Server（百度搜索），本地 @Tool + MCP 混合工具体系 |
| @Tool 规范 | 10 个本地 @Tool 全部返回 POJO DTO，框架自动序列化，不手动拼 JSON |
| 会话管理 | Redis StateStore + sessionId 多轮对话、聊天历史 DB 持久化（ChatSession + ChatMessage） |
| 上下文压缩 | CompactionConfig（阈值 8000 tokens）长对话自动压缩 |
| HITL 确认 | RequireUserConfirm + confirmTools 端点，用户确认后恢复 Agent 流 |
| Permission | Agent 工具级 ALLOW/DENY 控制，MCP 搜索工具与本地 @Tool 权限平等 |
| Token 安全 | JWT jti 黑名单机制、主动登出 |
| 配置管理 | 多环境配置（application.yml + application-dev.yml）、环境变量读取 API Key |
| 模型配置管理 | 前端管理 LLM Provider/API Key/模型名，支持全局默认和用户私有模型 |
| Docker 部署 | docker-compose 6 服务编排（MySQL + Redis + Milvus + etcd + MinIO + WebSearch） |
| 单元测试 | Security/Tool/Service/Controller 四层测试，含集成测试（IT） |

## 5. 面试故事线

面试时怎么讲这个项目：

"我做了一个 AI 面试诊断助手，核心是一个全能 Agent（HarnessAgent），覆盖简历诊断、模拟面试、面试复盘和知识检索 4 大核心环节。

技术上，整个系统基于 AgentScope Java v2 + Spring Boot 3（Servlet MVC 栈）。Agent 是单实例全能型设计——通过 SysPrompt 定义意图识别流程，根据用户问题类型自动选择相应工具。有 10 个本地 @Tool（解析简历、面试出题、回答分析、转写、知识检索等），联网搜索通过 MCP 协议接入百度搜索作为兜底。

认证方面，用 Spring Security + JWT 做无状态认证，Token 带 jti 唯一标识，登出时写入黑名单表。角色用 UserRole 枚举管理，分 USER 和 ADMIN 两种。所有接口统一返回 ApiResponse 格式，认证和授权异常由 GlobalExceptionHandler 统一处理。

RAG 方面，我用 Milvus 2.4.6 做向量存储，DashScope text-embedding-v3 做 Embedding。知识库是多租户架构——管理员创建公共库，用户创建私有库，Agent 检索时自动联合搜索。支持 4 种分块策略、异步入库管道、在线重建索引、多路召回 + RRF 融合 + Reranker 精排。

LLM 层面支持多 Provider 切换，用户可配置私有 API Key 和模型，优先级是用户私有 > 用户默认 > 全局默认 > application.yml 兜底。Embedding、转写、Rerank 都有独立 API Key 配置，与 LLM 解耦，切换模型不影响这些服务。

工具调用方面，10 个本地 @Tool 全部返回 POJO DTO，框架自动序列化。联网搜索通过 MCP 协议直连 OpenWebSearch MCP Server（百度引擎），McpClientBuilder 显式注册。Fallback 策略是确定性规则——知识库返回 total==0 或相关性不足时，SysPrompt 要求 Agent 必须调用 search 联网兜底。上下文管理用 CompactionConfig（8000 tokens）自动压缩。

前端用 React 19 + Ant Design 6，SSE 流式对话支持思考过程可见、HITL 人工确认（高危工具调用弹窗）、会话历史管理和全文搜索。有 8 个前端页面：登录、对话主页、简历诊断、分析报告、知识库管理主页、文档管理、文档详情、检索测试。

面试评分这块，我设计了 6 个评估维度，每个维度独立打分。

工程规范方面，所有实体继承 BaseEntity 审计基类。7 个枚举类消除硬编码。DTO 用 @Valid + JSR-303 校验。异常体系用 BusinessException 基类加子类，全局处理器统一响应。Agent 池用 Caffeine 有界缓存，最多 500 个实例，30 分钟不活跃自动淘汰。Redis 拆成 SearchResultCacheService 和 RateLimitService，职责单一。"

量化数据：17 张数据库表、10 个本地 @Tool（全部 POJO 返回）+ 1 个 MCP 联网搜索、7 个枚举类、6 个评估维度、4 种分块策略、6 个 Docker 基础设施容器、8 个前端页面、Caffeine Agent 池（500 上限）、Redis 双 Service（缓存+限流）、Token 黑名单机制。
