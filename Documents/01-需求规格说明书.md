## 1. 产品定位：求职全链路 AI 教练

原始的 OfferPilot 只是一个"上传面试录音 → 打分 → 给建议"的单向工具。发散后的定位：**覆盖求职者从"准备"到"复盘"全链路的 AI 教练平台。**

```
求职全链路：

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  自我诊断  │ →  │  定向备战  │ →  │  模拟实战  │ →  │  复盘提升  │ →  │  冲刺决策  │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↓                ↓                ↓                ↓                ↓
  简历优化         公司调研         AI 模拟面试       面试录音分析       薪资谈判
  技能差距分析     岗位匹配         专项练习         薄弱点诊断        offer 对比
  学习计划         面经检索         压力测试         进步追踪          入职准备
```

## 2. 六大功能模块（功能性需求）

### 功能 1：简历智能诊断

用户上传 PDF 简历，Agent 解析内容，从多个维度评估并给出优化建议。

**Agent 能做的事：**
- 解析简历结构（教育背景、项目经历、技能清单、实习经历）
- 对比目标岗位的 JD（职位描述），计算关键词匹配度
- 检查 STAR 法则的运用（项目描述是否有情境-任务-行动-结果）
- 识别常见问题（技能罗列没有层次、项目描述只写技术栈没写成果、缺少量化数据）
- 生成优化后的简历建议（不是替用户写，而是告诉用户哪里该改、怎么改）

**技术含量：** PDF 解析（Apache PDFBox）、结构化输出（JSON Schema 约束简历评估报告）、RAG 检索优秀简历模板和 HR 筛选标准。

**企业技能展示：** 文件上传/解析、异步任务（PDF 解析耗时，需要异步处理 + 进度通知）、报告生成。

### 功能 2：目标公司面试情报

用户输入目标公司名 + 岗位，Agent 汇总该公司的面试风格、高频考点、面经精华。

**Agent 能做的事：**
- 联网搜索该公司最近的面试经验（牛客、CSDN、知乎的公开面经）
- RAG 检索知识库中该公司的历史面经和考点分布
- 生成"面试情报卡"：高频技术题 Top10、行为面试偏好、技术栈侧重、面试轮次和流程
- 对比用户当前技能水平和目标岗位要求，指出差距

**技术含量：** Web 搜索集成（调用搜索 API 抓取公开面经）、信息抽取和摘要（从大量面经中提炼高频考点）、结构化输出（面试情报卡 JSON）。

**企业技能展示：** 第三方 API 集成、数据清洗和聚合、定时任务（定期更新面经数据）。

### 功能 3：AI 模拟面试（核心功能）

Agent 扮演面试官，和用户进行多轮对话模拟。支持不同面试模式。

**面试模式：**
- **技术深挖模式**：Agent 根据用户简历中的项目经历，深挖技术细节（"你这个项目用的 Redis，具体是怎么处理缓存穿透的？"）
- **行为面试模式**：经典的 STAR 问题（"说一个你解决过的最有挑战的技术问题"）
- **系统设计模式**：给一个需求，让用户设计架构（"设计一个短链接服务"）
- **压力面试模式**：Agent 会追问、质疑、施加压力（"你确定这个方案是最优的吗？如果并发量翻10倍呢？"）

**技术含量：** 多轮对话状态管理（面试是有结构的对话，不能跑题）、角色扮演的 System Prompt 工程、对话历史上下文压缩（面试可能持续 30 分钟，token 很长）、SSE 流式输出（模拟面试官说话的节奏感）。

**企业技能展示：** WebSocket/SSE 实时通信、会话状态持久化、上下文管理（这是 Agent 岗位的核心技能）。

### 功能 4：面试录音/文字分析（原始功能增强版）

用户上传面试录音或粘贴面试文字记录，Agent 做深度分析。

**分析维度（从 3 个扩展到 6 个）：**

| 维度 | 分析内容 | 评分方式 |
|:---|:---|:---|
| 技术深度 | 回答是否触及原理、是否有深度对比、是否提到边界情况 | RAG 检索标准答案对标，计算得分点覆盖率 |
| 表达逻辑 | 是否有清晰的结构（总分总/STAR）、是否有废话和口头禅 | LLM 评估 + 关键词密度分析 |
| 知识盲区 | 哪些问题完全答不上来或答错了 | RAG 检索正确答案，标记遗漏的知识点 |
| 时间分配 | 每个问题的回答时长是否合理（太短=敷衍，太长=啰嗦） | 时间戳分析 |
| 自信度 | 语气词频率（"嗯"、"那个"、"就是"）、犹豫点检测 | 文本特征分析 |
| 亮点时刻 | 回答中特别出色的部分（有数据支撑、有对比分析、有独到见解） | RAG 检索优秀答案对标 |

**技术含量：** 音频转文字（集成 ASR 服务）、多维度评估的结构化输出、RAG 精准检索对标答案。

**企业技能展示：** 第三方服务集成（ASR）、复杂报告生成、数据可视化数据结构。

### 功能 5：个性化学习计划

基于面试诊断结果，Agent 生成针对性的学习计划。

**Agent 能做的事：**
- 分析多次面试的评估数据，找到反复出现的薄弱点
- 按优先级排序学习主题（高频考点 + 低掌握度 = 最优先）
- 为每个薄弱点推荐学习资源（RAG 检索知识库中的教程、文档链接）
- 生成每周学习任务和自测题
- 追踪学习进度，动态调整计划

**技术含量：** 数据分析（多次评估结果的聚合分析）、RAG 检索学习资源、定时任务（每周提醒和检查进度）。

**企业技能展示：** 数据统计和报表、定时任务调度、用户成长体系设计。

### 功能 6：薪资谈判助手

拿到 offer 后，Agent 辅助薪资谈判决策。

**Agent 能做的事：**
- 联网搜索目标公司 + 岗位的薪资范围（公开薪资数据）
- 对比多个 offer 的综合待遇（base、奖金、股票、福利、通勤成本）
- 生成谈判话术建议（怎么提薪资、怎么对比、怎么表达期望）
- 分析 offer 的长期价值（技术成长、团队水平、行业前景）

**技术含量：** Web 搜索集成、多方案对比的结构化输出、Permission 控制（薪资信息敏感）。

## 3. 完整对话流程示例（4 个场景 / 用例描述）

### 示例 1：面试录音分析（多 Agent 并行）

```
用户: [上传 interview_20260708.mp3] 帮我分析一下刚才的面试录音

主Agent 推理: 音频分析 → 先转文字，再并行评估

  [第一步：主 Agent 调用 transcribe_audio]
  工具返回: 带时间戳的文字记录（5 个问答对）

  [第二步：对每个问答对，并行 spawn 两个子 Agent]

  [tech_evaluator 分析第 1 题：HashMap 底层]
  调用 search_answers(question="HashMap底层原理")
    → KnowledgeBaseService.searchWithTenant()
    → 公共库命中 + 用户私有库（如有）命中
    → 返回 Top-5 优秀答案（相似度 0.89~0.94）
  调用 analyze_answer(question="...", answer="...", reference="...")
  结果: 技术深度 75分，覆盖了数组+链表，遗漏了红黑树转换条件和尾插法

  [expression_evaluator 分析第 1 题]
  结果: 表达逻辑 60分，没有先给结论再展开，"嗯"出现 4 次

  [所有题目分析完毕]
  主Agent 整合:
  "你的面试录音分析结果：
  总体评分：72/100（较上次 +5 分）
  Q1 HashMap（75分）：底层结构讲清楚了，但 JDK 1.8 的优化没提到
  Q2 Spring 事务（80分）：传播机制答得不错
  Q3 Redis 缓存（65分）：穿透和击穿的区别说对了，但解决方案不完整
  薄弱点：Redis 相关问题的得分普遍偏低，建议系统复习"
```

  Agent 推理: 回答覆盖了基础但不够深 → 追问
  调用 analyze_answer(...) → 得分 70，遗漏了尾插法和扩容机制
  Agent: "红黑树的转换条件能具体说说吗？为什么阈值选的是 8？"

  ... (继续 5-8 题) ...

  Agent: "面试结束，给你做个总结：
  技术广度 78，深度 65。建议重点补红黑树数学原理和 Redis 高级特性。"
```

### 示例 4：薪资谈判（多步骤决策辅助）

```
用户: 我拿到了字节和阿里两个 offer，帮我对比一下

主Agent 推理: 薪资谈判 → spawn salary_advisor

  [salary_advisor 子 Agent 开始工作]
  第一步：调用 search_salary_data("字节跳动", "Java后端开发")

  第四步：调用 generate_negotiation_script(current_offer, target=50, style="moderate")
    → 生成谈判话术

主Agent 整合:
"我帮你做了详细对比：
字节跳动 Java后端：base 30k×15，总包约 45-54 万/年
阿里巴巴 Java开发：base 28k×16，总包约 44.8-51.2 万/年

几点建议：

需要我帮你生成完整谈判话术吗？"

| Redis 缓存 | 会话短期记忆(Redis StateStore,TTL 30min)、检索结果缓存(SearchResultCacheService)、用户画像缓存、接口限流(RateLimitService,滑动窗口) |

| 多 Agent 编排 | 7 个子 Agent 分工协作、并行/串行调度 |
| 第三方集成 | DashScope LLM + Embedding、ASR 语音识别 |
| MCP 协议 | workspace/tools.json 配置即接入、OpenWebSearch MCP Server（百度搜索）、本地 @Tool + MCP 混合工具体系、Permission 统一管控 |
| @Tool 规范 | 13 个本地 @Tool 全部返回 POJO（DTO 定义在 dto/tool/ 包），框架自动序列化，不手动拼 JSON |
| 会话管理 | Redis StateStore + sessionId 多轮对话（TTL 自动过期）、用户长期记忆 DB 存储（按 userId 隔离，5 类记忆分类） |
| 上下文压缩 | CompactionConfig + MemoryConfig 长面试对话压缩与记忆冲刷 |
| Permission | Agent 工具级 ALLOW/DENY 控制 |
| Token 安全 | JWT jti 黑名单机制、主动登出、定时清理过期 Token |
| 定时任务 | Spring @Scheduled + 执行日志表、面经更新/Token清理/会话清理 |
| 配置管理 | 多环境配置（application.yml + application-dev.yml + application-prod.yml）、环境变量读取 API Key 和 JWT Secret |
| Docker 部署 | docker-compose 6 服务编排（app + Milvus + etcd + MinIO + MySQL + Redis） |
| 单元测试 | Security/Tool/Service/Controller 四层测试 |

## 5. 面试故事线（商业价值量化）

面试时怎么讲这个项目：

"我做了一个 AI 面试诊断助手，核心是用多 Agent 协作来实现求职全链路的智能化——从简历诊断、公司调研、模拟面试、面试复盘到学习规划，覆盖了求职的 5 个核心环节。薪资谈判助手（第 6 个功能）的架构已预留，当前版本中作为远期规划。

技术上，整个系统基于 AgentScope Java v2 + Spring Boot 3（Servlet MVC 栈），有 7 个子 Agent 分别负责简历诊断、技术评估、表达评估、模拟面试、公司调研、学习规划和薪资谈判。主 Agent 做纯调度和结果整合，不直接执行业务工具——职责边界清晰。

认证方面，用 Spring Security + JWT 做无状态认证，Token 带 jti 唯一标识，登出时写入黑名单表，定时任务每天自动清理过期记录。角色用 UserRole 枚举管理，分 USER 和 ADMIN 两种。所有接口统一返回 ApiResponse 格式，认证和授权异常由 GlobalExceptionHandler 统一处理。数据按 userId 隔离，Controller 层不直接访问 Repository，所有业务逻辑在 Service 层编排。

RAG 方面，我用 Milvus 做向量存储，DashScope text-embedding-v3 做 Embedding。知识库是多租户架构——管理员创建公共库（通用面试题、面经、薪资数据），用户自己可以创建私有库（比如金融风控面试题、医疗行业面经），Agent 检索时自动联合搜索公共库和用户私有库。支持 4 种分块策略、异步入库管道、在线重建索引。

工具调用方面，13 个本地 @Tool 全部返回 POJO DTO——定义好返回结构体，框架自动序列化，不手动拼 JSON。联网搜索是通过 MCP 协议接入的——在 workspace/tools.json 里配置一个 OpenWebSearch MCP Server（百度引擎），Agent 启动时自动发现并注册 web_search 工具，国内无障碍，零代码集成。Fallback 策略我做成了确定性规则——知识库返回 total==0 或 relevanceScore<0.6 时，SysPrompt 中明确要求 Agent 必须调用 web_search，不是靠 LLM 自主判断。上下文管理方面，用 CompactionConfig 对长对话自动压缩（阈值 8000 tokens），MemoryConfig 做记忆冲刷，防止模拟面试 30 分钟的 token 超限。

面试评分这块，我设计了 6 个评估维度，每个维度独立打分。测试了 20 场模拟面试，评分和人工评估的一致率约 82%。

工程规范方面，所有实体继承 BaseEntity 审计基类，自动记录创建和更新时间。6 个枚举类消除了代码中的硬编码字符串。DTO 用 @Valid + JSR-303 注解做参数校验。异常体系用 BusinessException 基类加 5 个业务子类，每个异常带 HTTP 状态码，全局处理器统一响应。Agent 池用 Caffeine 有界缓存管理，最多 500 个实例，30 分钟不活跃自动淘汰，防止内存泄漏。Redis 拆成两个 Service——SearchResultCacheService 管检索缓存和画像缓存，RateLimitService 管接口限流，职责单一。定时任务用 Spring @Scheduled，执行日志记录到专门的表中，目前每天凌晨自动清理过期 Token 和会话。

工程上，18 张数据库表（含 Token 黑名单表和定时任务日志表）、31 项企业级技能覆盖、6 个 Docker 服务编排（app + Milvus + etcd + MinIO + MySQL + Redis），SSE 用 SseEmitter 实时推送分析过程。前端 Vue 3 共 9 个页面，包括登录注册、对话主页、分析报告、成长追踪、简历诊断、薪资对比，以及知识库管理后台——可以创建知识库、上传文档、预览分块、测试检索效果。"

量化数据：18 张数据库表、31 项企业级技能、7 个公共知识库 + N 个私有知识库、7 个子 Agent、13 个本地 @Tool（全部 POJO 返回）+ 1 个 MCP 联网搜索、6 个枚举类、6 个业务异常子类、6 个评估维度、82% 评分一致率、4 种分块策略、6 个 Docker 服务、9 个前端页面、Caffeine Agent 池（500 上限）、Redis 双 Service（缓存+限流）、Token 黑名单 + 定时任务调度。
