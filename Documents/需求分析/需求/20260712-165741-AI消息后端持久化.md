# AI 消息后端持久化

## 1. 背景与目标

### 背景

当前对话历史持久化机制存在不对称问题：

| 消息角色 | 保存方式 | 保存时机 | 可靠性 |
|---------|---------|---------|--------|
| USER | **后端同步保存** | SSE 流开始前，[ChatController.chatStream()](file:///opt/OfferPilot/src/main/java/com/tutorial/offerpilot/controller/ChatController.java#L127-L135) 直接调用 `chatHistoryService.saveMessage()` | 高（同步、数据库事务） |
| AI | **前端异步保存** | SSE 流结束后，[ChatPage.saveAiMessage()](file:///opt/OfferPilot/web/src/ui/pages/chat/ChatPage.tsx#L77-L106) 通过 HTTP POST 调用后端 API | 低（异步、3次重试后静默失败） |

**问题表现**：用户查看对话历史时，只有 USER 角色的消息，AI 角色的消息缺失。

### 根因

AI 消息的持久化完全依赖前端异步 HTTP 调用。以下任一情况都会导致 AI 消息丢失：

- 前端 `saveAiMessage` 回调中的 HTTP 请求因网络波动失败
- 用户在流结束后立即关闭页面，`onDone` 回调未执行
- JWT Token 在流期间过期，导致保存请求 401
- 浏览器标签页被后台挂起，异步请求被取消

### 目标

将 AI 消息的持久化逻辑从**前端异步**改为**后端同步**，在 SSE 流正常结束时由 `ChatController` 直接保存完整的 AI 消息到数据库，消除对前端网络请求的依赖。

## 4. 技术方案

### 整体思路

在 `ChatController` 的两个流方法（`chatStream` / `confirmTools`）中，累积 AI 响应内容并在流结束时同步保存到数据库。关键挑战在于 **HITL 确认会跨方法暂停/恢复流**，缓冲区内容需要在 `chatStream` → `confirmTools`（及可能的多轮嵌套 HITL）之间传递。

数据流向：

```
chatStream()
  ├── 创建 AiMessageBuffer
  ├── doOnNext 累积 delta/thinking/toolCall
  │     │
  │     ├── [普通流] AGENT_END / doOnComplete → saveAiMessage() → 完成
  │     │
  │     └── [HITL 触发] REQURE_USER_CONFIRM
  │              └── 将当前 buffer 镜像存入 ConfirmContext → 流暂停
  │
  └── confirmTools() 被调用
         ├── 从 ConfirmContext 恢复 buffer，继续累积
         ├── doOnNext 累积 delta/thinking/toolCall
         │     │
         │     ├── [普通流] AGENT_END / doOnComplete → saveAiMessage() → 完成
         │     │
         │     └── [嵌套 HITL] REQURE_USER_CONFIRM
         │              └── 更新 ConfirmContext 中的 buffer 镜像 → 流再次暂停
         │                     → 下一个 confirmTools() 继续以上流程
```

### 关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 保存时机 | `saveMessage` → `safeSend("done")` | 先持久化再通知前端，确保 DB 写入成功后才标记流结束 |
| 空内容保护 | `contentBuffer` 为空时跳过保存 | 避免写入无意义的空消息 |
| HITL 缓冲区传递 | 扩展 `ConfirmContext` 记录，存储字符串副本 | 线程安全、可跨方法访问 |
| 保存方法封装 | 提取 `saveAiMessage(String sessionId, String userId, AiMessageBuffer)` 私有方法 | chatStream / confirmTools / doOnComplete 三处复用 |

### 核心数据结构

#### AiMessageBuffer（缓冲区载体）

```java
/** AI 响应内容缓冲区，跨 HITL 边界传递时使用字符串副本 */
private record AiMessageBuffer(
        String content,       // 已累积的正文
        String thinking,      // 已累积的思考过程
        List<String> toolCalls // 已累积的工具调用名列表
) {
    AiMessageBuffer() { this("", "", new ArrayList<>()); }

    AiMessageBuffer appendContent(String delta) {
        return new AiMessageBuffer(content + delta, thinking, toolCalls);
    }
    AiMessageBuffer appendThinking(String delta) {
        return new AiMessageBuffer(content, thinking + delta, toolCalls);
    }
    AiMessageBuffer addToolCall(String name) {
        List<String> updated = new ArrayList<>(toolCalls);
        updated.add(name);
        return new AiMessageBuffer(content, thinking, updated);
    }
}
```

> **设计说明**：使用不可变 record + 字符串副本而非 `StringBuilder`，因为 HITL 场景下缓冲区需要跨方法边界传递，存储到 `ConcurrentHashMap<ConfirmContext>` 的必须是线程安全的不可变对象。每次追加返回新实例的开销极小（几 KB 字符串拼接）。

#### 扩展 ConfirmContext

```java
// 原定义
private record ConfirmContext(HarnessAgent agent, RuntimeContext ctx, String sessionId) {}

// 扩展后
private record ConfirmContext(
        HarnessAgent agent,
        RuntimeContext ctx,
        String sessionId,
        AiMessageBuffer buffer  // ← 新增：HITL 暂停时保存已累积内容
) {}
```

#### 4.1 私有辅助方法 `saveAiMessage()`

提取为 `ChatController` 的私有方法，三处复用：

```java
private void saveAiMessage(String sessionId, String userId, AiMessageBuffer buffer) {
    String content = buffer.content();
    if (content == null || content.isBlank()) {
        return; // 空内容跳过
    }
    try {
        MessageSaveRequest req = new MessageSaveRequest();
        req.setRole("AI");
        req.setContent(content);
        String thinking = buffer.thinking();
        if (thinking != null && !thinking.isEmpty()) {
            req.setThinkingContent(thinking);
        }
        List<String> toolCalls = buffer.toolCalls();
        if (toolCalls != null && !toolCalls.isEmpty()) {
            req.setToolCalls(objectMapper.writeValueAsString(toolCalls));
        }
        chatHistoryService.saveMessage(sessionId, userId, req);
        log.info("Saved AI message: sessionId={}, contentLen={}", sessionId, content.length());
    } catch (Exception e) {
        log.warn("Failed to save AI message: sessionId={}, error={}", sessionId, e.getMessage());
    }
}
```

### 保存时机对照

| 场景 | 旧方案（前端） | 新方案（后端） |
|------|-------------|-------------|
| 正常流 AGENT_END | 前端 onDone → HTTP POST | 后端 save → safeSend("done") |
| doOnComplete 兜底 | 前端 onDone → HTTP POST | 后端 save → safeSend("done") |
| HITL 确认后恢复流 | 前端 onDone → HTTP POST | 后端 save（含 pre+post 完整内容）→ safeSend("done") |
| 嵌套 HITL 恢复流 | 前端 onDone → HTTP POST | 后端 save（含全部多轮累积内容）→ safeSend("done") |
| 空响应 | 前端 saveAiMessage 空返回跳过 | 后端 saveAiMessage 空内容跳过 |
| 流异常/错误 | 不保存 | 不保存 |
| 客户端断开 | 不保存 | 不保存（doOnCancel 不触发保存） |
