# OfferPilot 系统核心流程矫正 — 流式思考可见性改造

> **文档版本**：v1.1

> **关联文档**：`20260712-111133-系统降配架构简化.md`（当前架构基准）、`20260711-025753-子Agent联网搜索兜底.md`（MCP 调用链路）

## 1. 背景与目标

### 1.1 背景

当前系统核心对话流程存在两个关键缺陷：

**缺陷一：LLM 思考过程不可见（"黑盒等待"）**

AgentScope v2 框架在执行过程中会发出丰富的事件流，包括：
- `THINKING_BLOCK_START / DELTA / END` — LLM 的推理链（reasoning chain）
- `TOOL_CALL_START / DELTA / END` — 工具调用状态（搜索知识库、联网搜索等）
- `TEXT_BLOCK_START / DELTA / END` — 最终回复文本

但 `ChatController` 仅透传了 `TEXT_BLOCK_DELTA` → SSE `delta` 事件，丢弃了 `THINKING_BLOCK_DELTA` 和 `TOOL_CALL_START`。前端也只能处理 `delta` 事件。

结果：用户发送问题后，看到的是长时间的 "AI 思考中..." 占位文字，然后突然出现完整回复。体验差，用户不知道系统在做什么，等待焦虑。

**缺陷二：核心流程语义不明确**

当前 System Prompt 虽然覆盖了工具使用场景，但缺乏明确的"分步执行"指令。Agent 可能跳过检索直接回复、或在知识库无结果时不做联网兜底。正确的流程应该是：

```
用户提问 → 意图识别 → 知识库召回 → [不足时] MCP 联网搜索 → LLM 整合 → 流式输出
```

### 1.2 目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | 思考过程实时可见 | THINKING_BLOCK_DELTA → SSE `thinking` 事件，前端展示关键摘要 |
| G2 | 工具调用状态可见 | TOOL_CALL_START → SSE `tool_call` 事件，告知用户"正在检索知识库/联网搜索" |
| G3 | 流程分步约束 | System Prompt 强制"先检索、后整合、不足时联网"的决策链 |
| G4 | 全场景覆盖 | 简历诊断、模拟面试、通用对话均享受相同改造 |

### 1.3 核心术语

| 术语 | 说明 |
|------|------|
| 思考块 (Thinking Block) | AgentScope 框架中 LLM 推理过程的流式输出，包含决策理由、工具选择分析等 |
| 工具调用事件 (Tool Call) | 框架通知 Agent 正在调用某个工具（如 search_questions、MCP search） |
| SSE 事件流 | Server-Sent Events 实时推送，当前支持 session / delta / done / error，矫正后新增 thinking / tool_call |
| 意图识别 | LLM 自主判断用户查询属于 practice（刷题）/ learn（学习）/ company（公司情报）/ salary（薪资）/ general（通用） |
| MCP 联网兜底 | 知识库（Milvus + MySQL）无结果时，LLM 自主决策调用 MCP web-search 工具 |

### 4.2 后端变更详解

#### 4.2.1 ChatController — 新增事件透传

当前 `chatStream()` 仅在 `doOnNext` 中处理两种事件：

```java
// 当前（仅透传文本和结束事件）
if (event.getType() == AgentEventType.TEXT_BLOCK_DELTA
        && event instanceof TextBlockDeltaEvent delta) {
    safeSend.accept("delta", delta.getDelta());
} else if (event.getType() == AgentEventType.AGENT_END) {
    safeSend.accept("done", "completed");
}
```

矫正后新增 5 种事件处理：

```java
// 矫正后（完整透传 7 种事件）
if (event.getType() == AgentEventType.THINKING_BLOCK_START) {
    safeSend.accept("thinking_start", "{}");
} else if (event.getType() == AgentEventType.THINKING_BLOCK_DELTA
        && event instanceof ThinkingBlockDeltaEvent delta) {
    safeSend.accept("thinking", delta.getDelta());
} else if (event.getType() == AgentEventType.THINKING_BLOCK_END) {
    safeSend.accept("thinking_end", "{}");
} else if (event.getType() == AgentEventType.TOOL_CALL_START
        && event instanceof ToolCallStartEvent toolCall) {
    safeSend.accept("tool_call", toolCall.getToolCallName());
} else if (event.getType() == AgentEventType.TOOL_CALL_END) {
    safeSend.accept("tool_call_end", "{}");
} else if (event.getType() == AgentEventType.TEXT_BLOCK_DELTA
        && event instanceof TextBlockDeltaEvent delta) {
    safeSend.accept("delta", delta.getDelta());
} else if (event.getType() == AgentEventType.AGENT_END) {
    safeSend.accept("done", "completed");
}
```

**SSE 事件语义定义：**

| SSE 事件名 | 触发时机 | 数据内容 | 前端行为 |
|-----------|---------|---------|---------|
| `session` | 流开始 | sessionId 字符串 | 存储会话 ID |
| `thinking_start` | LLM 开始推理 | `{}` | 展开/新建思考区域 |
| `thinking` | LLM 推理增量 | reasoning delta 文本 | 追加到思考区，折叠显示 |
| `thinking_end` | LLM 推理结束 | `{}` | 收起思考区，显示摘要 |
| `tool_call` | 工具调用开始 | 工具名称（如 `search_questions`） | 显示"正在搜索面试题库…" |
|※ `tool_call_end` | 工具调用结束 | `{}` | 将工具状态翻转为"已完成"，清除 loading |
| `delta` | 最终回复增量 | 回复文本片段 | 追加到 AI 消息正文 |
| `done` | 流结束 | `"completed"` | 标记完成，停止 loading |
| `error` | 异常 | 错误信息 | 显示错误提示 |

#### 4.2.2 AgentFactory — System Prompt 重写

当前 System Prompt 偏重能力描述，缺乏显式的分步执行约束。矫正后增加：

```text
【执行流程（必须严格遵守）】
对于用户的每个问题，你必须按以下步骤执行：

步骤 1：意图识别
  - 分析用户问题，判断属于哪种类型：practice（刷题）/ learn（学习）/ company（公司情报）
    / salary（薪资）/ general（通用）
  - 在思考中输出意图分类结果

步骤 2：知识库检索（仅检索型查询执行此步骤）
  - 适用于：知识查询、面试题搜索、公司信息、薪资数据等需要外部数据的场景
  - 优先使用 smart_search 或对应的专项搜索工具（search_questions/search_answers/
    search_resources）检索本地知识库
  - 若检索到足够信息 → 跳至步骤 4
  - 不适用于：模拟面试出题（generate_next_question）、回答分析（analyze_answer）、
    简历评估（evaluate_resume）等交互型操作，这些直接执行对应工具即可

步骤 3：联网搜索兜底（仅检索型查询执行此步骤）
  - 若知识库无结果或结果不充分 → 使用 search 工具（MCP 联网搜索）获取最新信息

步骤 4：信息整合
  - 将检索结果整合为自然语言回复
  - 引用来源（标注来自知识库/联网搜索）
  - 如有必要，对信息进行甄别和筛选

步骤 5：流式输出
  - 以清晰、结构化、易读的方式输出最终回复
```

### 7.2 风险与注意事项

| 风险 | 缓解措施 |
|------|--------|
| THINKING_BLOCK_DELTA 内容过长影响前端性能 | 前端做 max-height + overflow scroll 限制；`appendThinkingContent` 做字符数上限（建议 10000 字符），超出后停止追加；后端不做截断 |
| 某些模型可能不输出 THINKING_BLOCK 事件 | ChatController 用 instanceof 安全判断，无此事件时静默跳过 |
| SSE 事件顺序交错（thinking/delta 交替出现） | 前端按事件类型分流到不同缓冲区，不混入正文 content |
| 多轮 ReAct 循环（thinking→tool_call→thinking→tool_call→…） | 前端支持多轮交错：每次 `tool_call` 追加到 toolCalls 链，`tool_call_end` 清除 currentToolCall；thinking 内容持续追加到 thinkingContent |
| Tool Call 开始但未结束（异常中断） | 前端 Tool Call 状态在收到 done/error 时自动清除 |

### 7.3 向后兼容

新增的 SSE 事件类型（`thinking_start`/`thinking`/`thinking_end`/`tool_call`/`tool_call_end`）对老版本前端透明：当前 `processBuffer` 的 `else` 分支会静默跳过未识别的事件类型，不会报错。前端升级后即可消费新事件。

### 7.4 测试策略

| 测试类型 | 范围 | 方法 |
|---------|------|------|
| 后端单元测试 | ChatController 事件透传 | Mock `agent.streamEvents()` 返回含 THINKING_BLOCK_DELTA / TOOL_CALL_START / TOOL_CALL_END 的 Flux，验证 SseEmitter 输出对应 SSE 事件 |
| 前端联调测试 | chatService 事件解析 | 构造模拟 SSE 流（含全部新事件类型），验证 StreamCallbacks 各回调正确触发 |
