# 知识库文档管理 API 补齐

## 1. 背景与目标

### 背景

KnowledgeBaseController 当前仅实现 3 个基础端点（创建/列表/删除知识库），前端管理页面（`/admin/knowledge/:kbId` 文档列表页、`/admin/knowledge/:kbId/search` 检索测试页）已开发完成并配置了完整的 API 调用（`web/src/infra/constants.ts` 中定义了 8 个 KB 相关路径常量），但因后端缺少对应 Controller 端点，前端页面返回 404，无法联调。

### 目标

补齐 KnowledgeBaseController 的 8 个缺失端点，使前端知识库管理页面的文档管理、检索测试功能完整可用。

### 范围边界

| 范围 | 说明 |
|------|------|
| **做** | 在 KnowledgeBaseController 新增 8 个 REST 端点；在 KnowledgeBaseService 新增对应的 7 个业务方法 |
| **不做** | 新增 Entity/Repository/数据库表变更；前端代码修改；Milvus Collection 自动创建逻辑 |

### 关键术语

| 术语 | 说明 |
|------|------|
| KnowledgeBase (KB) | 知识库实体 (`kb_knowledge_base`)，对应一个 Milvus Collection |
| Document | 知识库下的文档 (`kb_document`)，上传后经异步入库管道最终索引到向量库 |
| Chunk | 文档分块 (`kb_chunk`)，文档经分块后每个 chunk 对应 Milvus 一条向量 |
| Ingestion Pipeline | 异步入库管道：PARSING → CHUNKING → EMBEDDING → INDEXING → ACTIVE |
| Reindex | 重新索引，将文档状态重置为 UPLOADED 后重新触发入库管道 |

## 2. 功能清单

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|--------|--------|------|
| 知识库统计 | GET `/api/v1/kb/{kbId}/stats` | P0 | 返回知识库名称、文档总数、分块总数、活跃/失败文档数 |
| 文档管理 | GET `/api/v1/kb/{kbId}/docs` | P0 | 分页查询知识库下的文档列表 |
| 文档管理 | POST `/api/v1/kb/{kbId}/docs` | P0 | 上传文档文件（multipart/form-data），创建文档记录并触发异步入库 |
| 文档管理 | GET `/api/v1/kb/{kbId}/docs/{docId}` | P0 | 查询文档详情，含分块预览列表 |
| 文档管理 | DELETE `/api/v1/kb/{kbId}/docs/{docId}` | P0 | 删除文档及其关联分块（DB + Milvus），更新知识库统计 |
| 文档管理 | GET `/api/v1/kb/{kbId}/docs/{docId}/progress` | P0 | 轮询文档入库进度（status + progress 百分比） |
| 文档管理 | POST `/api/v1/kb/{kbId}/docs/{docId}/reindex` | P1 | 重置文档状态为 UPLOADED 并重新触发异步入库管道 |
| 检索测试 | POST `/api/v1/kb/{kbId}/search` | P1 | 向量检索测试，返回 Top-K 结果及耗时 |

## 4. 技术方案

### 整体架构

```
前端 kbService.ts (已就绪，12 个方法)
    ↓ HTTP
KnowledgeBaseController (新增 8 个端点)
    ↓ 扁平化参数
KnowledgeBaseService (新增 7 个方法)
    ↓ Entity/Repository
JPA Repository (已就绪)
    ↓
MySQL 8.0 (表结构完整)

上传流程：
Controller → 保存文件 → 创建 KbDocument(UPLOADED) 
    → DocumentIngestionService.ingestDocument(docId) [@Async]
    → PARSING → CHUNKING → EMBEDDING → INDEXING → ACTIVE

检索流程：
Controller → KnowledgeBaseService.searchKb(kbId, query, topK)
    → VectorSearchService.search(collectionName, query, topK)
    → Milvus
```

### 关键设计决策

1. **不新建 DocumentController**：文档是知识库的子资源，路由层级为 `/api/v1/kb/{kbId}/docs/**`，统一放入 KnowledgeBaseController 管理，符合 RESTful 资源嵌套规范。

2. **Service 方法参数扁平化**：遵循 Service 层规范，KnowledgeBaseService 新增的方法参数使用基础类型（String kbId、String docId 等），不传递 DTO 对象。

3. **文件上传复用 MinIO**：文档文件上传后存储到 MinIO（通过已有的 FileUploadService），文件路径写入 `KbDocument.filePath`，后续入库管道从 MinIO 读取文件内容。

4. **入库进度轮询**：前端每 2 秒轮询 `GET /docs/{docId}/progress`，后端直接从 `KbDocument.status` 和 `KbDocument.progress` 字段返回。

5. **检索性能**：VectorSearchService 已封装 Milvus search，Controller 直接透传结果，增加 `latencyMs` 字段供前端展示。

```java
// 1. 知识库统计
@GetMapping("/{kbId}/stats")
public ApiResponse<KbStatsResponse> getKbStats(
        @PathVariable String kbId, @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Get KB stats: kbId={}", kbId);
    return ApiResponse.success(kbService.getKbStats(kbId, getUserId(currentUser), currentUser));
}

// 2. 文档列表
@GetMapping("/{kbId}/docs")
public ApiResponse<List<DocResponse>> listDocs(
        @PathVariable String kbId, @AuthenticationPrincipal UserDetails currentUser) {
    log.info("List docs: kbId={}", kbId);
    return ApiResponse.success(kbService.listDocs(kbId, getUserId(currentUser), currentUser));
}

// 3. 上传文档
@PostMapping("/{kbId}/docs")
public ResponseEntity<ApiResponse<DocResponse>> uploadDoc(
        @PathVariable String kbId,
        @RequestParam("file") MultipartFile file,
        @RequestParam(defaultValue = "AUTO") String chunkStrategy,
        @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Upload doc: kbId={}, fileName={}, strategy={}", kbId, file.getOriginalFilename(), chunkStrategy);
    // ... save file → create KbDocument → trigger ingestion
}

// 4. 文档详情
@GetMapping("/{kbId}/docs/{docId}")
public ApiResponse<DocDetailResponse> getDocDetail(
        @PathVariable String kbId, @PathVariable String docId,
        @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Get doc detail: kbId={}, docId={}", kbId, docId);
    return ApiResponse.success(kbService.getDocDetail(kbId, docId, getUserId(currentUser), currentUser));
}

// 5. 删除文档
@DeleteMapping("/{kbId}/docs/{docId}")
public ResponseEntity<Void> deleteDoc(
        @PathVariable String kbId, @PathVariable String docId,
        @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Delete doc: kbId={}, docId={}", kbId, docId);
    kbService.deleteDoc(kbId, docId, getUserId(currentUser), currentUser);
    return ResponseEntity.noContent().build();
}

// 6. 入库进度
@GetMapping("/{kbId}/docs/{docId}/progress")
public ApiResponse<Map<String, Object>> getDocProgress(
        @PathVariable String kbId, @PathVariable String docId,
        @AuthenticationPrincipal UserDetails currentUser) {
    // 返回 { status: "INDEXING", progress: 80 }
}

// 7. 重新索引
@PostMapping("/{kbId}/docs/{docId}/reindex")
public ApiResponse<Void> reindexDoc(
        @PathVariable String kbId, @PathVariable String docId,
        @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Reindex doc: kbId={}, docId={}", kbId, docId);
    kbService.reindexDoc(kbId, docId, getUserId(currentUser), currentUser);
    return ApiResponse.success();
}

// 8. 检索测试
@PostMapping("/{kbId}/search")
public ApiResponse<SearchTestResponse> search(
        @PathVariable String kbId,
        @RequestBody @Valid SearchTestRequest req,
        @AuthenticationPrincipal UserDetails currentUser) {
    log.info("Search KB: kbId={}, query={}, topK={}", kbId, req.getQuery(), req.getTopK());
    return ApiResponse.success(kbService.searchKb(kbId, req.getQuery(), req.getFilterExpr(), req.getTopK(), getUserId(currentUser), currentUser));
}
```

## 5. 数据库变更

**无。** 现有表结构已完整：

- `kb_knowledge_base` — kbId, name, description, category, visibility, status, milvusCollection, ownerId, documentCount, chunkCount
- `kb_document` — docId, kbId, fileName, filePath, fileType, fileSize, chunkCount, chunkStrategy, status, progress, errorMessage, metadataJson, tags, uploadedAt, indexedAt
- `kb_chunk` — docId, kbId, chunkIndex, content, contentHash, tokenCount, milvusOffset
