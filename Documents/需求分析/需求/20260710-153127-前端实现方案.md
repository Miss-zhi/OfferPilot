# OfferPilot 前端实现方案

## 1. 背景与目标

### 1.1 背景

OfferPilot 当前已完成 Spring Boot 3 后端全部功能开发（18 张数据库表、7 个子 Agent、13 个 @Tool、完整 REST API），但**暂无前端界面**，所有功能仅可通过 Swagger/Postman 调用，不具备产品化的用户交互能力。

### 1.2 目标

为 OfferPilot 构建完整的前端 SPA 应用，覆盖求职全链路的 6 大功能模块，提供与后端 API 无缝对接的用户界面，使项目具备完整的"前后端分离"架构形态和产品化演示能力。

### 1.3 核心术语

| 术语 | 说明 |
|------|------|
| SPA | 单页应用（Single Page Application），前端路由由浏览器端控制 |
| SSE | 服务端推送事件（Server-Sent Events），用于 AI 对话的流式响应 |
| JWT | JSON Web Token，无状态认证方案，前端存储在 localStorage |
| 知识库管理后台 | 管理员/用户对 Milvus 向量知识库进行 CRUD 管理的界面 |
| 对话主页 | 用户与 AI Agent 进行多轮对话的核心交互页面 |

### 4.3 关键设计决策

#### 4.3.1 路由守卫（AuthGuard）

```
用户访问任意页面
    ↓
检查 localStorage 中是否有 JWT Token
    ├── 无 Token → 跳转 /login
    └── 有 Token
        ↓
    检查 Token 是否过期（解析 exp 字段）
        ├── 已过期 → 清除 Token + 跳转 /login
        └── 未过期 → 渲染目标页面
```

实现方式：React Router 6 的 `<Navigate>` + 自定义 `AuthGuard` 组件包裹需认证路由。

#### 4.3.2 SSE 流式对话（Chat 页面核心）

```
用户输入消息 → 点击发送
    ↓
chatService.sendMessage(message, sessionId)
    ↓
fetchEventSource('/api/v1/offerpilot/chat/stream', {
    method: 'POST',
    headers: { Authorization: 'Bearer ' + token },
    body: JSON.stringify({ message, sessionId }),
    onmessage(event) {
        switch (event.event) {
            case 'TEXT_BLOCK_DELTA':
                // 逐字追加 AI 回复文本 → MarkdownRenderer 实时渲染
                break;
            case 'TOOL_CALL_START':
                // 显示"正在分析..."加载状态
                break;
            case 'TOOL_CALL_END':
                // 隐藏加载状态
                break;
            case 'AGENT_END':
                // 对话结束，更新消息列表
                break;
        }
    }
})
```

技术要点：
- 使用 `@microsoft/fetch-event-source` 库（比原生 EventSource 支持 POST 请求）
- SSE 流式文本通过 `SseStreamRenderer` 组件逐字追加，Markdown 实时渲染
- 对话历史存储在 `chatStore` 中，支持跨页面恢复

#### 4.3.3 知识库文档进度轮询

文档上传后异步入库（PARSING → CHUNKING → EMBEDDING → INDEXING → ACTIVE），前端通过轮询展示进度：

```
上传成功后 → 启动轮询（每 2 秒）
    ↓
GET /api/v1/admin/kb/{kbId}/docs/{docId}/progress
    ↓
解析 { status, progress } 
    ├── status === 'ACTIVE' → 停止轮询，刷新文档列表
    ├── status === 'FAILED' → 停止轮询，显示错误信息
    └── 其他 → 更新进度条，继续轮询
```

#### 4.3.4 六维度雷达图（面试分析报告）

使用 ECharts 雷达图展示面试评分的 6 个维度：

```
技术深度(75)  表达逻辑(60)  知识覆盖(70)
自信度(68)    时间管理(80)  亮点时刻(78)
```

实现方式：封装 `ScoreRadarChart` 组件，接收 `dimensions` 数据，渲染 ECharts 雷达图。

#### 4.3.5 开发代理配置

Vite 开发服务器通过代理将 `/api` 请求转发到 Spring Boot 后端：

```typescript
// vite.config.ts
export default defineConfig({
  server: {
    port: 3000,
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
      },
    },
  },
});
```

这样前端开发时无需处理 CORS，生产环境通过 Nginx 反向代理解决。

### 4.4 分层架构

遵循 `.qoder/rules/web-layer.mdc` 定义的 4 层架构：

```
app → ui/pages → service → infra
                → ui/components
```

| 层 | 职责 | 禁止 |
|----|------|------|
| app | 路由配置、AuthProvider 全局认证状态、Ant Design ConfigProvider 主题 | 不放业务代码 |
| ui/pages | 页面渲染、用户交互、调用 service 层方法 | 不直接发 HTTP 请求 |
| ui/components | 通用可复用组件（聊天气泡、文件上传器、图表等） | 不包含特定业务逻辑 |
| service | API 调用封装、请求参数类型、响应类型定义 | 不引用 UI 组件 |
| infra | Axios 实例、请求/响应拦截器、Token 管理 | 不含业务逻辑 |
| store | Zustand 全局状态（auth/chat/kb），跨页面共享 | 不包含 UI 渲染逻辑 |

### 6.7 响应拦截器规范

遵循 `.qoder/rules/web-page-layer.mdc`，`infra/http.ts` 响应拦截器自动解包后端 `ApiResponse<T>` 格式：

```typescript
// 后端返回：{ code: 200, message: "success", data: { ... } }
// 拦截器解包后 Service 层直接拿到 data：{ ... }
http.interceptors.response.use(
  response => {
    const body = response.data;
    if (body && typeof body === 'object' && 'code' in body) {
      if (body.code === 200) {
        response.data = body.data;
      } else {
        return Promise.reject(new Error(body.message || '请求失败'));
      }
    }
    return response;
  }
);
```

**全部接口遵循 RESTful 规范**（参考 `.qoder/rules/api-restful.mdc`）：小写字母 + 中划线命名，标准 HTTP 方法映射。

## 7. 实施计划

| 阶段 | 内容 | 预计页面数 | 预计工时 |
|------|------|-----------|---------|
| **一期** | 项目初始化 + 登录注册 + 对话主页 + 分析报告 + 简历诊断 | 4 页 | 5-7 天 |
