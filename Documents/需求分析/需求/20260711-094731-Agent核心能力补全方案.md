# Agent 核心能力补全 — MCP 联网搜索、多 Agent 架构、SalaryTool 完善

## 1. 背景与目标

### 背景

OfferPilot 第 v5.0 版设计文档（[02-系统架构设计说明书](../02-系统架构设计说明书.md)）定义了完整的 Agent 能力体系，包括 13 个 @Tool + MCP 联网搜索 + 1主7子多 Agent 协作。当前实现与设计文档存在 3 处缺口：

| 缺口 | 当前状态 | 设计目标 |
|------|---------|---------|
| MCP 联网搜索 | `workspace/tools.json` 缺失；AgentFactory 未配置 workspace 路径 | Agent 可通过 `web_search` 工具获取互联网实时信息 |
| 多 Agent 协作 | 单 HarnessAgent，无 SubagentDeclaration | 1 主 Agent（调度中心）+ 7 子 Agent（专业分工），支持并行 |
| SalaryTool @Tool 不完整 | 仅 `search_salary` 一个 @Tool | 3 个 @Tool：`search_salary_data` + `compare_offers` + `generate_negotiation_script` |

### 目标

1. 补全 MCP 联网搜索能力，Agent 可实时获取互联网信息（作为知识库检索的低相关度 fallback）
2. 实现 1 主 + 7 子 Agent 协作架构，子 Agent 按白名单继承主 Toolkit 的工具
3. SalaryTool 新增 2 个 @Tool，覆盖完整薪资谈判链路

### 术语表

| 术语 | 说明 |
|------|------|
| MCP (Model Context Protocol) | AgentScope v2 支持的工具接入协议，通过 `tools.json` 配置远程工具 |
| SubagentDeclaration | AgentScope v2 中定义子 Agent 的声明式配置（name、sysPrompt、tools 白名单） |
| Toolkit 过滤副本 | 子 Agent spawn 时从主 Toolkit 按白名单复制，与其他子 Agent 隔离 |
| @Tool | AgentScope v2 本地工具注解，标注在 Spring Bean 方法上 |

## 4. 技术方案

### 4.1 整体架构变更

```
当前（单 Agent）：
  Controller → AgentFactory → 1 个 HarnessAgent → 11 个 @Tool

目标（多 Agent）：
  Controller → AgentFactory → 主 HarnessAgent（调度中心）
    ├── .workspace("./workspace") → 扫描 tools.json → web_search
    ├── .subagent(resume_coach)      → toolkit 过滤副本: [parse_resume, evaluate_resume, search_questions]
    ├── .subagent(tech_evaluator)    → toolkit 过滤副本: [search_answers, analyze_answer, search_questions]
    ├── .subagent(expression_evaluator) → toolkit 过滤副本: [analyze_answer]
    ├── .subagent(mock_interviewer)  → toolkit 过滤副本: [generate_next_question, search_answers, analyze_answer]
    ├── .subagent(company_researcher)→ toolkit 过滤副本: [search_company_interviews, search_questions]
    ├── .subagent(study_planner)     → toolkit 过滤副本: [track_progress, search_resources, search_questions]
    └── .subagent(salary_advisor)    → toolkit 过滤副本: [search_salary, compare_offers, generate_negotiation_script]
```

### 4.2 MCP 联网搜索 — tools.json

新建 `workspace/tools.json`：

```json
{
  "mcpServers": {
    "web-search": {
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "open-web-search-mcp"],
      "env": {
        "SEARCH_ENGINE": "baidu",
        "MAX_RESULTS": "5"
      }
    }
  }
}
```

AgentFactory 在 `HarnessAgent.builder()` 中添加：

```java
.workspace(Path.of("./workspace"))  // 自动扫描 tools.json → 注册 web_search
```

HarnessAgent 启动时自动扫描 `workspace/tools.json`，连接 MCP Server，将 `web_search` 注册为自身工具——无需在 Java 代码中手动注册。

### 4.4 主 Agent SysPrompt — 重写为调度中心

```java
private String buildSystemPrompt() {
    return """
        You are OfferPilot 面试诊断助手的调度中心。

        【你的唯一职责】理解用户需求，分派任务给子 Agent，整合结果回复用户。
        【严禁行为】你绝对不能直接调用任何业务工具（search_questions、analyze_answer、
                   parse_resume 等）。你的工具箱中只有 spawn/resume_subagent 和
                   web_search。所有业务操作必须通过子 Agent 完成。

        子 Agent 分派指南：
        - 简历分析/优化 → spawn resume_coach
        - 面试回答分析/评分 → spawn tech_evaluator + expression_evaluator（并行）
        - 模拟面试练习 → spawn mock_interviewer
        - 公司面试情报 → spawn company_researcher
        - 学习计划/进度 → spawn study_planner
        - 薪资查询/offer对比/谈判策略 → spawn salary_advisor
        - 通用知识问答（无匹配子Agent时） → 调用 web_search 从互联网获取信息

        调度规则：
        1. 你只能使用 spawn（创建子Agent任务）和 resume（恢复子Agent任务）。
        2. 子 Agent 返回结果后，用自然语言整合回复用户，不暴露内部调度过程。
        3. 如果用户意图跨越多个子Agent领域，按优先级依次 spawn。
        4. 绝不绕过子Agent直接调用业务工具——即使你知道答案也要走子Agent流程。

        IMPORTANT — Tool output interpretation:
        - When you call `generate_next_question`, it returns guidance (not a ready question).
        - When you call `analyze_answer`, it returns the raw Q&A plus evaluation guidance.
        - When you call `evaluate_resume`, it returns the resume text plus evaluation guidance.
        - For all guidance-based tools, never echo the raw guidance to the user.
        """.stripIndent();
}
```

### 4.6 AgentFactory.buildAgent() 最终结构

```java
private HarnessAgent buildAgent(String userId) {
    log.info("Building HarnessAgent for userId={}", userId);

    // 1. 构建 Toolkit（13 个 @Tool + registerMetaTool）
    Toolkit toolkit = buildToolkit();

    // 2. 按优先级选择模型
    Model model = resolveModel(userId);

    // 3. 系统提示词（调度中心）
    String sysPrompt = buildSystemPrompt();

    // 4. 中间件
    TokenMonitorMiddleware tokenMonitor = new TokenMonitorMiddleware();
    CostControlMiddleware costControl = new CostControlMiddleware();

    // 5. Permission
    PermissionContextState perms = buildPermissions();

    // 6. 定义 7 个子 Agent
    SubagentDeclaration resumeAgent = buildResumeAgent();
    SubagentDeclaration techEvalAgent = buildTechEvalAgent();
    SubagentDeclaration exprEvalAgent = buildExprEvalAgent();
    SubagentDeclaration mockAgent = buildMockAgent();
    SubagentDeclaration companyAgent = buildCompanyAgent();
    SubagentDeclaration studyAgent = buildStudyAgent();
    SubagentDeclaration salaryAgent = buildSalaryAgent();

    // 7. 构建 HarnessAgent
    HarnessAgent agent = HarnessAgent.builder()
            .name("offerpilot_" + userId)
            .sysPrompt(sysPrompt)
            .model(model)
            .toolkit(toolkit)
            .workspace(Path.of("./workspace"))     // ← 新增：MCP 扫描路径
            .subagent(resumeAgent)                 // ← 新增
            .subagent(techEvalAgent)               // ← 新增
            .subagent(exprEvalAgent)               // ← 新增
            .subagent(mockAgent)                   // ← 新增
            .subagent(companyAgent)                // ← 新增
            .subagent(studyAgent)                  // ← 新增
            .subagent(salaryAgent)                 // ← 新增
            .permissionContext(perms)              // ← 新增
            .middleware(tokenMonitor)
            .middleware(costControl)
            .enablePlanMode()
            .enableTaskList()
            .build();

    log.info("HarnessAgent built: name={}, model={}", agent.getName(), model);
    return agent;
}
```

### 4.7 SalaryTool 新增 @Tool 方法

#### 现状分析：DTO 孤岛问题

`dto/tool/` 下 13 个 DTO 中有 2 个"孤岛"——仅被 REST API 引用，没有任何 @Tool 返回它们：

| DTO | 当前使用者 | Agent 可调用？ |
|-----|-----------|:---:|
| `OfferComparisonResult` | `SalaryController`、`SalaryService` | ❌ |
| `NegotiationScriptResult` | `SalaryController`、`SalaryService` | ❌ |

本次新增 2 个 @Tool 方法后，两个孤岛 DTO 将接入 Agent 体系，补齐 SalaryTool 的完整薪资谈判链路：

```
SalaryTool
├── search_salary               → SalarySearchResult       ✅ 已有
├── compare_offers              → OfferComparisonResult    ← 新增
└── generate_negotiation_script → NegotiationScriptResult  ← 新增
```

#### 实现模式对比

两个新增 @Tool 方法采用**不同模式**，原因在于 AgentScope `@ToolParam` 仅支持基本类型 + `String`：

| @Tool 方法 | 参数复杂度 | 模式 | 说明 |
|-----------|-----------|------|------|
| `search_salary`（已有） | String × 2 | **简单委托** | 直接透传参数给 Service |
| `generate_negotiation_script` | String × 2 + double × 1 | **简单委托**（同 `search_salary`） | 构建简单 DTO → 调 Service |
| `compare_offers` | 嵌套列表 `List<OfferItem>` | **JSON 适配**（需额外步骤） | LLM 传 JSON 字符串 → `ObjectMapper` 解析 → 调 Service |

#### 代码实现

**① `generate_negotiation_script` — 简单委托模式（与 `search_salary` 一致）**

```java
@Tool(name = "generate_negotiation_script",
      description = "根据当前 offer 和目标期望薪资，生成薪资谈判话术建议。")
public NegotiationScriptResult generateScript(
        @ToolParam(name = "current_offer",
                   description = "当前 offer 的文本描述，如'字节跳动 30k×15 + 期权'")
        String currentOffer,
        @ToolParam(name = "target_salary",
                   description = "目标期望年薪（万），如 50")
        double targetSalary,
        @ToolParam(name = "negotiation_style",
                   description = "谈判风格：assertive（强硬）/ moderate（温和）/ conservative（保守），默认 moderate")
        String negotiationStyle
) {
    log.info("generate_negotiation_script called: targetSalary={}, style={}",
             targetSalary, negotiationStyle);
    NegotiationScriptRequest req = new NegotiationScriptRequest();
    req.setCurrentOffer(currentOffer);
    req.setTargetSalary(targetSalary);
    req.setNegotiationStyle(
            negotiationStyle != null ? negotiationStyle : "moderate");
    return salaryService.generateNegotiationScript(req);
}
```

**② `compare_offers` — JSON 适配模式**

`SalaryService.compareOffers(OfferCompareRequest)` 的参数是 `List<OfferItem>` 嵌套对象，超出 @Tool 参数类型限制。因此 Agent 以 JSON 字符串传参，Tool 内部解析后调用 Service：

```java
private final ObjectMapper objectMapper;  // 构造器注入

@Tool(name = "compare_offers",
      description = "对比多个 offer 的综合待遇。输入多个 offer 详情 JSON 数组，生成对比分析报告和推荐建议。")
public OfferComparisonResult compareOffers(
        @ToolParam(name = "offers_json", description = """
                多个 offer 详情 JSON 数组。每个 offer 包含：
                - company: 公司名称
                - position: 岗位名称
                - base: 月薪（K）
                - months: 月数（如 15）
                - bonus: 奖金信息（如 "3-6个月"）
                - stock: 股票/期权信息
                - location: 工作地点
                示例: [{"company":"字节跳动","position":"Java后端","base":30,"months":15}]
                """)
        String offersJson
) {
    log.info("compare_offers called");
    OfferCompareRequest req = parseOfferJson(offersJson);
    return salaryService.compareOffers(req);
}

/** 将 JSON 字符串解析为 OfferCompareRequest */
private OfferCompareRequest parseOfferJson(String json) {
    try {
        return objectMapper.readValue(json, OfferCompareRequest.class);
    } catch (Exception e) {
        log.warn("Failed to parse offers JSON", e);
        throw new BusinessException(400,
                "offer JSON 格式错误: " + e.getMessage());
    }
}
```

**注意**：`SalaryService` 无需改动——JSON 解析在 Tool 层完成，Service 保持已有的 `OfferCompareRequest` 签名不变。

## 7. 影响范围与风险评估

| 风险项 | 等级 | 缓解措施 |
|--------|------|---------|
| 多 Agent 架构增加 token 消耗 | 中 | 子 Agent 独立 session，不受主 Agent compaction 影响；Caffeine 缓存复用 Agent 实例 |
| MCP Server 依赖外部 npx 进程 | 低 | `tools.json` 可随时移除禁用；AgentScope 框架自带进程生命周期管理 |
| compare_offers JSON 解析失败 | 低 | LLM 通常能生成合法 JSON；解析失败抛出 `BusinessException`，由全局异常处理器统一响应 |
| SalaryService 签名不匹配 | 无 | `SalaryService` 无需改动——JSON 解析在 Tool 层完成，Service 保持 `OfferCompareRequest` 签名 |
| 子 Agent 工具白名单遗漏 | 低 | 参照设计文档逐一定义；集成测试覆盖所有子 Agent 路径 |
