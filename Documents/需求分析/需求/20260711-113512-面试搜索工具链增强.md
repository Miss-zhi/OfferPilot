# 面试搜索工具链全面增强

## 1. 背景与目标

### 背景

当前 OfferPilot 面试搜索工具链存在以下结构性不足：

1. **搜索精度单一**：4 个 `search_*` 工具仅基于 Milvus 向量检索 + DB LIKE 回退，缺少 BM25 关键词匹配和元数据过滤能力
2. **RAG 与联网断层**：知识库无结果时不会自动联网兜底，依赖 LLM 在子 Agent 层面自行判断调用 `web_search` MCP，不可靠且延迟高
3. **搜索无个性化**：所有用户搜同一关键词得到相同结果，未利用 `KnowledgeMastery` 薄弱点做加权排序
4. **搜索架构分散**：4 个工具各自独立遍历 PUBLIC 知识库，未复用已有的 `searchMultiCollection` 方法，无全局排序
5. **无搜索反馈闭环**：无法追踪哪些搜索结果被采纳、哪些查询高频零结果，缺少数据驱动的内容建设依据
6. **`web_search` MCP 未下沉**：仅 3 个子 Agent 可调用，主 Agent 和基础搜索工具链无法受益

### 目标

将搜索工具链从"关键词 → 向量检索 → 回退"的简单管道，升级为"意图理解 → 多路召回 → 融合排序 → 个性化加权 → 自动兜底"的智能搜索体系。

### 非目标

- 不涉及前端搜索 UI 改造（工具层变更对前端透明）
- 不引入新的搜索引擎（复用已有 Bing via `web-search` MCP）
- 不替换 Milvus 为其他向量数据库

## 2. 功能清单

### P0 — 快速见效（~4h）

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|--------|--------|------|
| 搜索精度 | 多 Collection 合并检索 | P0 | 替换逐个 KB 遍历，使用已有的 `VectorSearchService.searchMultiCollection` 统一召回+全局排序 |
| RAG 兜底 | web_search 下沉至基础搜索链 | P0 | `KnowledgeBaseService` 4 个搜索方法内增加 MCP 联网兜底：Milvus→DB→MCP |
| 搜索缓存 | 高频查询 Redis 缓存 | P0 | 相同关键词 5 分钟内直接返回缓存，降低 Embedding API 调用和 Milvus 压力 |

### P1 — 搜索质量（~10h）

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|--------|--------|------|
| 搜索精度 | 混合检索（Vector + BM25） | P1 | Milvus 2.4 原生 Hybrid Search，同时利用语义相似度和关键词精确匹配 |

| 搜索智能 | Query 改写/扩展 | P1 | LLM 将短查询扩展为多条检索短语，分别检索后去重合并（如"Java并发"→["Java并发面试题","线程池原理","volatile关键字"]） |

### P2 — 智能个性（~12h）

| 功能模块 | 功能点 | 优先级 | 说明 |

| 反馈闭环 | 搜索反馈追踪 | P2 | 新增 `search_feedback` 表，通过回话上下文隐式判断用户是否采纳搜索结果 |
| 反馈闭环 | 零结果分析 + 热门搜索 | P2 | 记录无结果查询、高频搜索词，驱动知识库内容建设 |
| 新能力 | JD 解析搜索 | P2 | 上传职位描述 → 提取关键技能 → 自动搜索匹配面试题和学习资源 |

## 4. 技术方案

### 4.1 整体架构

```
用户查询
    │
    ▼
[SmartSearchTool] / [4×独立 Tool]
    │
    ▼
┌─────────────────────────────────────────────────┐
│              SearchMiddleware（P2）               │
│  Query 改写 → 意图分类 → 多路召回 → 融合排序      │
│  → 个性化加权 → MCP 兜底                          │
└─────────────────────────────────────────────────┘
    │                │                │
    ▼                ▼                ▼
 Milvus           MySQL            MCP
(向量+BM25)    (LIKE回退)    (web_search兜底)
```

### 4.2 P0：多 Collection 合并 + MCP 兜底 + Redis 缓存

**核心改动文件**：`KnowledgeBaseService.java`

```
现有流程（逐个遍历）:
  for (KbKnowledgeBase kb : kbs) {
      hits = vectorSearch(kb.collection, keyword, 10);  // 每个 KB 独立查 10 条
      items.addAll(convert(hits));
  }
  // 无全局排序，不同 KB 的结果混在一起

P0 流程（合并检索 + 兜底）:
  collections = kbs.stream().map(Kb::getCollection).toList();
  hits = searchMultiCollection(collections, keyword, 10, 20);  // 全局排序取 Top-20
  items = convert(hits);

  if (items.isEmpty()) {
      items = dbFallback(keyword);  // MySQL LIKE
  }
  if (items.isEmpty()) {
      items = mcpFallback(keyword);  // web_search MCP
  }
```

**Redis 缓存**：
- Key: `search:{tool}:{keyword}`，TTL 5 分钟
- 任意搜索命中即回填缓存
- 文档入库/删除时通过 `@CacheEvict` 失效对应知识库缓存

### 4.3 P1：混合检索 + 结构化查询

**Milvus Hybrid Search**（`VectorSearchService.hybridSearch`）：

```java
// Milvus 2.4 支持 AnnSearchRequest + Reranker 实现混合检索
public List<SearchHit> hybridSearch(String collection, String query, int topK) {
    float[] vector = embeddingService.embed(query);
    
    // 向量检索请求
    AnnSearchReq vectorReq = AnnSearchReq.builder()
            .annsField("vector").data(vector).topK(topK * 2).build();
    
    // BM25 关键词检索请求（Milvus 2.4 原生支持）
    AnnSearchReq keywordReq = AnnSearchReq.builder()
            .annsField("sparse_vector").data(sparseEncode(query)).topK(topK * 2).build();
    
    // WeightedRanker 加权融合
    HybridSearchReq hybridReq = HybridSearchReq.builder()
            .collectionName(collection)
            .searchRequests(List.of(vectorReq, keywordReq))
            .ranker(new WeightedRanker(0.7f, 0.3f))  // 向量权重0.7，BM25权重0.3
            .topK(topK)
            .build();
    
    return milvusClient.hybridSearch(hybridReq);
}
```

**结构化查询参数** — 搜索工具 DTO 升级：

```java
// 旧：单 keyword
searchQuestions(String keyword)

// 新：结构化 SearchRequest
searchQuestions(SearchRequest req) {
    // req.keywords: "Java 并发"    ← 主查询词
    // req.category: "专业技能"     ← Milvus filter 过滤
    // req.difficulty: "hard"       ← Milvus filter 过滤
    // req.company: "字节跳动"       ← 公司过滤
    // req.position: "后端开发"      ← 岗位过滤
    // req.topK: 10                 ← 返回数量
}
```

### 4.4 P2：个性化 + 统一入口 + 反馈闭环

**薄弱点加权**（`SearchMiddleware`）：

```
读取 KnowledgeMastery WHERE userId = ? AND score < 60
    → 提取薄弱知识点列表
    → 对搜索结果中匹配薄弱点的 item.relevanceScore *= 1.3
```

**Query 意图分类**（轻量规则 + LLM 兜底）：

```
输入: "字节跳动后端面试一般问什么"
    → 规则匹配: 含"公司"关键词 → 意图=公司调研
    → 策略: search_company_interviews + search_questions 联合检索
    → 结果标记 source=kb/db/web

输入: "帮我出几道Java并发题"
    → 规则匹配: 含"出题"关键词 → 意图=刷题
    → 策略: search_questions(difficulty=medium) + 返回 via mock_interview 流程
```

**搜索反馈闭环**：

```
Agent 对话中隐式判断:
- 用户说"这个答案很好" → feedback.helpful = true
- 用户说"不是我要的，换个关键词" → feedback.helpful = false
- 搜索零结果 → log.zero_result = true → 纳入内容建设待办
```

## 5. 数据库变更

```sql
-- 搜索反馈表
CREATE TABLE op_search_feedback (
    id              BIGINT          PRIMARY KEY AUTO_INCREMENT  COMMENT '主键',
    user_id         VARCHAR(64)     NOT NULL                    COMMENT '用户ID',
    query_text      VARCHAR(500)    NOT NULL                    COMMENT '原始查询词',
    tool_name       VARCHAR(64)     NOT NULL                    COMMENT '调用的工具名',
    result_source   VARCHAR(32)     NOT NULL                    COMMENT '结果来源: kb/db/web',
    result_count    INT             NOT NULL DEFAULT 0           COMMENT '返回结果数',
    helpful         TINYINT         DEFAULT NULL                COMMENT '是否有用: 1=有用 0=无用 NULL=未知',
    session_id      VARCHAR(64)     DEFAULT NULL                COMMENT '关联会话ID',
    created_at      TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    INDEX idx_user_id (user_id),
    INDEX idx_query_text (query_text(100)),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='搜索反馈记录';

-- 搜索日志表
CREATE TABLE op_search_log (
    id              BIGINT          PRIMARY KEY AUTO_INCREMENT  COMMENT '主键',
    user_id         VARCHAR(64)     DEFAULT NULL                COMMENT '用户ID（匿名搜索可为空）',
    query_text      VARCHAR(500)    NOT NULL                    COMMENT '原始查询词',
    expanded_queries TEXT           DEFAULT NULL                COMMENT 'Query扩展后的多个检索词（JSON数组）',
    tool_name       VARCHAR(64)     NOT NULL                    COMMENT '调用的工具名',
    intent          VARCHAR(32)     DEFAULT NULL                COMMENT '意图分类: practice/learn/company/salary',
    milvus_hits     INT             NOT NULL DEFAULT 0           COMMENT 'Milvus命中数',
    db_hits         INT             NOT NULL DEFAULT 0           COMMENT 'DB回退命中数',
    web_hits        INT             NOT NULL DEFAULT 0           COMMENT 'MCP联网命中数',
    total_results   INT             NOT NULL DEFAULT 0           COMMENT '最终返回结果数',
    zero_result     TINYINT         NOT NULL DEFAULT 0           COMMENT '是否零结果: 1=是',
    latency_ms      BIGINT          NOT NULL DEFAULT 0           COMMENT '搜索耗时（毫秒）',
    created_at      TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    INDEX idx_user_id (user_id),
    INDEX idx_query_text (query_text(100)),
    INDEX idx_zero_result (zero_result),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='搜索日志';
```

## 6. 接口设计

### 6.1 结构化搜索（改造现有）

```
POST /api/v1/kb/search/questions
POST /api/v1/kb/search/answers
POST /api/v1/kb/search/company-interviews
POST /api/v1/kb/search/resources
```

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| keywords | String | 是 | 搜索关键词，多个用空格分隔 |
| category | String | 否 | 分类过滤：专业技能/项目经验/情景分析/行为面试/职业规划 |
| difficulty | String | 否 | 难度过滤：easy/medium/hard |
| company | String | 否 | 公司名称过滤 |
| position | String | 否 | 岗位名称过滤 |
| topK | Integer | 否 | 返回数量，默认 10，最大 50 |

**响应示例：**

```json
{
  "total": 5,
  "source": "kb",
  "items": [
    {
      "questionId": "q-abc123",
      "content": "请描述 Java 中 volatile 关键字的作用和原理",
      "category": "专业技能",
      "difficulty": "medium",
      "relevanceScore": 0.92,
      "source": "kb"
    }
  ]
}
```
