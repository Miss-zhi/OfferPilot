# Harness 演示提示词清单

> 以下每个条目按「概念 + 演示操作」两段式组织，直接复制演示操作输入 Qoder 即可生成对应资产。

---

## 一、缺失 Rule（8 个）

### 1. code-complexity.mdc

**概念：** 控制代码复杂度，防止 AI 生成过深的嵌套逻辑和超长方法。

**演示操作：**
```
在 .qoder/rules/ 下创建 code-complexity.mdc，要求：方法体不超过 300 行，嵌套层级不超过 5 层，单个类不超过 1000 行。触发条件为 always_on，适用于所有 Java 和 TypeScript 文件。
```

### 2. code-security-basics.mdc

**概念：** 基础安全约束，防止 AI 在代码中硬编码敏感信息或引入常见安全漏洞。

**演示操作：**
```
在 .qoder/rules/ 下创建 code-security-basics.mdc，要求：禁止硬编码密码/密钥/Token，禁止拼接 SQL，禁止在日志中输出敏感信息，数据库连接使用环境变量或配置中心。触发条件为 always_on，适用于所有代码文件。
```

### 3. react-component.mdc

**概念：** 约束 React 组件的编写规范，确保组件职责单一、结构清晰。

**演示操作：**
```
在 .qoder/rules/ 下创建 react-component.mdc，要求：单个组件文件不超过 200 行，组件只接收必要的 props 并定义 TypeScript interface，禁止在组件内直接操作 DOM，事件处理函数以 handle 前缀命名。触发条件为 glob 匹配 *.tsx 和 *.jsx 文件。
```

### 4. react-hooks.mdc

**概念：** 约束 React Hooks 的使用规范，防止依赖遗漏和滥用。

**演示操作：**
```
在 .qoder/rules/ 下创建 react-hooks.mdc，要求：自定义 Hook 以 use 前缀命名，useEffect 必须显式声明依赖数组，禁止在循环/条件/嵌套函数中调用 Hook，useMemo 和 useCallback 仅在确有性能问题时使用。触发条件为 glob 匹配 *.tsx 和 *.jsx 文件。
```

### 5. db-pagination.mdc

**概念：** 约束数据库分页查询规范，防止全表扫描和内存溢出。

**演示操作：**
```
在 .qoder/rules/ 下创建 db-pagination.mdc，要求：列表查询接口必须支持分页，默认 pageSize 不超过 50，使用 MyBatis-Plus 的 Page 对象封装分页参数，禁止在业务层做全量查询后再内存分页。触发条件为 model_decision，描述为"涉及列表查询或分页时激活"。
```

### 6. db-transaction.mdc

**概念：** 约束数据库事务使用规范，确保数据一致性。

**演示操作：**
```
在 .qoder/rules/ 下创建 db-transaction.mdc，要求：涉及多表写操作的方法必须添加 @Transactional 注解，事务注解加在 Service 层而非 Controller 层，只读方法使用 @Transactional(readOnly = true)，禁止在事务中进行远程调用。触发条件为 model_decision，描述为"涉及多表写操作或事务管理时激活"。
```

### 7. project-layer-dependency.mdc

**概念：** 定义项目专属的分层依赖方向，防止跨层引用和反向依赖。

**演示操作：**
```
在 .qoder/rules/ 下创建 project-layer-dependency.mdc，基于当前 Harness 工程的组件化架构：api 层只能调用 business 层和 component 接口层，business 层只能调用 component 接口层，component 实现层只能调用 domain 层，domain 层不依赖任何上层。禁止 controller 直接调用 mapper，禁止 component 之间互相依赖。触发条件为 always_on。
```

### 8. project-db-compat.mdc

**概念：** 约束数据库兼容性规范，确保 SQL 在不同环境下可移植。

**演示操作：**
```
在 .qoder/rules/ 下创建 project-db-compat.mdc，要求：禁止使用数据库特有函数（如 MySQL 的 GROUP_CONCAT），日期处理使用 Java 标准库而非 SQL 函数，建表语句使用通用类型（VARCHAR/BIGINT/DATETIME），禁止在 SQL 中使用反引号包裹字段名。触发条件为 model_decision，描述为"涉及 SQL 编写或数据库操作时激活"。
```

## 二、缺失 Skill（6 个）

### 1. create-proposal

**概念：** 生成功能需求设计文档，按标准模板输出背景、功能清单、涉及模块、技术方案、数据库变更、接口设计，分类更细，便于追溯。

**演示操作：**
```
/create-skill create-proposal，功能：根据用户输入的需求描述，自动生成结构化的需求设计文档，包含背景、功能清单、涉及模块、技术方案、数据库变更、接口设计。文件命名规则为 日期-时间-需求名.md，存储路径为 Documents/需求分析/需求/。
```

### 2. create-bugfix

**概念：** 引导 AI 按标准流程定位和修复 Bug，确保修复可追溯。

**演示操作：**
```
/create-skill create-bugfix，功能：根据用户描述的 Bug 现象，按"定位→分析→修复→验证"四步流程执行，输出修复记录和变更文件清单。文件命名规则为 日期-时间-bugfix-简述.md，存储路径为 Documents/需求分析/BugFix/。
```

### 3. create-refactor

**概念：** 引导 AI 按标准流程执行代码重构，确保重构安全可控。

**演示操作：**
```
/create-skill create-refactor，功能：根据用户指定的重构目标，按"分析现状→制定方案→逐步重构→编译验证"四步流程执行，每步完成后验证编译通过。文件命名规则为 日期-时间-refactor-简述.md，存储路径为 Documents/需求分析/重构/。
```

### 4. create-component

**概念：** 按 java-component-layer.mdc 规范生成标准组件骨架。

**演示操作：**
```
/create-skill create-component，功能：根据用户输入的组件名称，按 java-component-layer.mdc 规范生成标准组件骨架，包含 interfaces/implementations 两层结构和 pom.xml
```

### 5. create-service-test

**概念：** 为指定 Service 方法生成服务层测试类，连真实数据库但不启动 Web 容器，基于 BaseServiceTest 基类。

**演示操作：**
```
/create-skill create-service-test，功能：根据用户指定的 Service 类名和方法名 + 测试场景，自动生成 JUnit 服务测试类。读取目标 Service 方法定义，分析方法签名和返回值，使用 BaseServiceTest 基类（提供 Factory 工厂方法和断言辅助），连接真实数据库但不启动 Web 容器（@SpringBootTest(webEnvironment = NONE)）。测试数据使用 99xxx 开头的 ID。输出到 test/java/{模块}/service/ 目录。文件命名规则为 日期-时间-service-test-简述.md，存储路径为 Documents/需求分析/测试/。
```

### 6. create-integration-test

**概念：** 为指定业务链路生成集成测试三件套（case.json + setup.sql + verify.sql），启动完整 Web 应用执行。

**演示操作：**
```
/create-skill create-integration-test，功能：根据用户指定的 Service 类名和方法名 + 测试场景描述，自动生成集成测试三件套文件。读取目标 Service 方法定义、扫描数据依赖，生成 case.json（目标+方法+参数，ID 用 99xxx 开头）、setup.sql（先清理后插入，保证幂等）、verify.sql（每个检查点 SELECT 一行，actual vs expected）。使用 GenericMethodRunner 反射调用 Service 方法，启动完整 Web 应用（@SpringBootTest(webEnvironment = RANDOM_PORT)）。输出到 cases/{Service}/{Method}/{场景}/ 目录。文件命名规则为 日期-时间-integration-test-简述.md，存储路径为 Documents/需求分析/测试/。
```

## 三、缺失 Agent（3 个）

### 1. dba-agent

**概念：** 数据库专家 Agent，专门处理 SQL 审查、索引优化和数据库设计问题。

**演示操作：**
```
/create-subagent dba-agent，职责：审查 SQL 语句的性能和规范性，检查索引设计是否合理，优化慢查询，审查建表语句是否符合项目规范。
```

### 2. test-runner-agent

**概念：** 测试执行 Agent，统一编排编译→执行→验证→报告四阶段流程，发现失败自动定位并建议修复方案。

**演示操作：**
```
/create-subagent test-runner-agent，职责：按编译依赖→执行测试→验证结果→生成报告四阶段执行测试。编译阶段运行 mvn compile，失败则报错；执行阶段逐个运行三件套用例，记录状态和耗时；验证阶段比对 verify.sql 的 actual vs expected；报告阶段汇总用例名称 / 状态 / 失败原因 / 差异。失败时定位到具体代码行并给出修复建议。
```

### 3. audit-agent

**概念：** 安全审查 Agent，专门检查代码中的安全漏洞和敏感信息泄露。

**演示操作：**
```
/create-subagent audit-agent，职责：扫描代码中的硬编码密钥/密码，检查 SQL 注入风险，审查接口权限控制是否完整，检查日志中是否输出敏感信息。
```
