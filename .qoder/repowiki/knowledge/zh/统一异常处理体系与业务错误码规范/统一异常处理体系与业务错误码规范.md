## 1. 采用的系统/方案
- 基于 Spring Boot 的 `@RestControllerAdvice` + `@ExceptionHandler` 全局异常拦截机制。
- 自定义业务异常继承自统一的 `BusinessException`，通过 errorCode 字段承载 HTTP 状态码级别的错误码。
- 所有 API 响应统一封装为 `ApiResponse<T>`，包含 code、message、data 三个字段。

## 2. 核心文件与包
- 基础响应体：`common/ApiResponse.java`
- 全局异常处理器：`exception/GlobalExceptionHandler.java`
- 业务异常基类：`exception/BusinessException.java`
- 领域异常子类：
  - `exception/ResourceNotFoundException.java`（404）
  - `exception/InvalidCredentialsException.java`（401）
  - `exception/DuplicateUserException.java`（409）
  - `exception/AccountDisabledException.java`（403）
  - `exception/FileUploadException.java`（400）
  - `exception/RateLimitException.java`（429）

## 3. 架构与约定
- **异常层次**：`BusinessException` 作为根，各具体场景异常仅负责绑定固定 HTTP 状态码，不再携带额外字段。Service 层直接抛出这些异常，Controller 层不捕获业务异常。
- **全局映射**：`GlobalExceptionHandler` 中按优先级注册处理器：
  - `BusinessException` → 使用其 errorCode 作为 HTTP 状态码，返回 `ApiResponse.error(code, message)`；日志级别 warn。
  - `MethodArgumentNotValidException` → 聚合 Bean Validation 字段错误信息，返回 400。
  - `IllegalArgumentException` → 视为编程错误，返回 500。
  - 兜底 `Exception` → 记录 error 日志并返回 500。
- **响应格式**：成功路径由 Controller 或 Service 调用 `ApiResponse.success(data)` / `ApiResponse.success()`；失败路径一律经全局处理器包装为 `ApiResponse.error(...)`，保证前端只面对单一结构。
- **日志策略**：业务异常用 warn 级别记录，非法参数与未知异常用 error 级别记录堆栈，便于运维排查。

## 4. 开发者应遵循的规则
1. **不要吞掉业务异常**：在 Service 层遇到可预期的业务错误时，直接 throw 对应的 `*Exception`，不要在 Controller 里 try-catch 再手动构造 ApiResponse。
2. **新增业务错误先扩展异常类**：如果现有异常无法表达新的 HTTP 语义，新建一个继承 `BusinessException` 的子类并在其中绑定默认 errorCode，然后在 `GlobalExceptionHandler` 中增加对应 `@ExceptionHandler`。
3. **禁止在 Controller 中返回裸异常**：所有异常最终都会落到全局处理器，避免重复包装导致嵌套 ApiResponse。
4. **Bean Validation 错误信息保持可读性**：使用 `@NotBlank`、`@Size` 等注解声明约束，让 `MethodArgumentNotValidException` 自动聚合错误消息，无需在代码中拼接。
5. **对外错误码即 HTTP 状态码**：当前实现将 errorCode 直接映射到 HTTP 状态码，新增错误码时应选择恰当的 4xx/5xx 值，避免滥用 500。
6. **不要依赖 `IllegalStateException` 等运行时异常做业务分支**：它们会被全局处理器当作“服务器内部错误”返回 500，掩盖真实问题。
