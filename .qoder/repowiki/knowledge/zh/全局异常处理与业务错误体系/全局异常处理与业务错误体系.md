## 1. 采用的系统/方案
- 基于 Spring Boot 的 `@RestControllerAdvice` + `@ExceptionHandler` 实现**全局异常统一拦截**。
- 定义一个核心业务异常基类 `BusinessException`，所有领域异常均继承自它，通过 errorCode（HTTP 状态码）+ message 描述错误。
- 统一的响应体 `ApiResponse<T>` 承载 code/message/data，由全局处理器包装后返回给前端。
- 对参数校验、非法参数、未知异常分别做差异化处理并记录日志。

## 2. 关键文件与包
- 全局处理器：`src/main/java/com/tutorial/offerpilot/exception/GlobalExceptionHandler.java`
- 业务异常基类：`src/main/java/com/tutorial/offerpilot/exception/BusinessException.java`
- 具体业务异常：
  - `ResourceNotFoundException`（404）
  - `InvalidCredentialsException`（401）
  - `AccountDisabledException`（账号禁用）
  - `DuplicateUserException`（重复用户）
  - `FileUploadException`（400）
  - `RateLimitException`（429）
- 统一响应体：`src/main/java/com/tutorial/offerpilot/common/ApiResponse.java`
- 典型抛出点（示例）：`AuthService`、`ChatController`、`KnowledgeBaseService`、`ModelConfigService`、`FileService` 等

## 3. 架构与约定
- **异常层次**：`RuntimeException` → `BusinessException` → 各语义化子类。业务层只抛这些异常，不向上抛原始第三方异常。
- **错误码即 HTTP 状态码**：`BusinessException` 的 errorCode 直接作为 HTTP 状态码返回；未显式指定时默认 400。
- **统一响应格式**：成功走 `ApiResponse.success(...)`，失败走 `ApiResponse.error(code, message)`，字段仅包含 code/message/data，且使用 `NON_NULL` 过滤空字段。
- **全局处理器职责**：
  - `BusinessException` → 按 errorCode 返回对应状态码 + 错误消息
  - `MethodArgumentNotValidException` → 聚合字段校验错误，返回 400
  - `IllegalArgumentException` → 视为“意外”入参问题，返回 500 并记录 error 日志
  - 兜底 `Exception` → 捕获一切未预期异常，返回 500 并记录堆栈
- **日志策略**：业务异常用 warn 级别记录 code/message，未知异常用 error 记录完整堆栈，便于线上排查。

## 4. 开发者应遵循的规则
1. **不要直接抛 `RuntimeException` / `Exception`**：在 service/controller 中遇到可预期的业务错误，一律抛 `BusinessException` 或其子类。
2. **优先使用语义化异常子类**：如找不到资源抛 `ResourceNotFoundException`，认证失败抛 `InvalidCredentialsException`，限流抛 `RateLimitException`，上传失败抛 `FileUploadException`。
3. **错误码与 HTTP 状态码保持一致**：构造 `BusinessException` 时传入的 errorCode 即为对外 HTTP 状态码，避免自行在 Controller 里设置 ResponseStatus。
4. **不要在业务层吞掉异常再返回 ApiResponse**：让 `GlobalExceptionHandler` 统一包装，保持调用链简洁。
5. **参数校验交给 Bean Validation**：配合 `@Valid`/`@Validated`，由全局处理器自动聚合为 400 响应。
6. **对外暴露的消息要友好**：message 面向前端/客户端，不应泄露内部堆栈或敏感信息。
7. **新增业务异常时**：新建继承 `BusinessException` 的子类并固定 errorCode，同时在 `GlobalExceptionHandler` 中确认已有兜底处理，无需额外注册 handler。
