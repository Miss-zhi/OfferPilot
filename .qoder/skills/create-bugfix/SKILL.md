# create-bugfix

## 目的

根据用户描述的 Bug 现象，按标准化四步流程（定位→分析→修复→验证）执行 Bug 修复，确保修复过程可追溯、不引入新问题，并输出修复记录文档和变更文件清单。

## 前置规范

**必须先读取并严格遵循**以下规则文件：

1. `.qoder/rules/md-naming-format.mdc` — MD 文档命名规范（yyyyMMdd-HHmmss-{name}.md）
2. `.qoder/rules/java-early-return.mdc` — Java 代码规范（修复代码时需遵循）
3. `.qoder/rules/java-exception.mdc` — Java 异常处理规范（修复异常相关 Bug 时参考）
4. `.qoder/rules/copyright-header.mdc` — 版权注释规范（新文件和修改文件均需检查）

**同时必须遵循以下记忆约束：**
- 修改前需先向用户说明发现的问题、原因分析和解决方案，获得确认后再执行
- 修复文档仅记录完整过程，不进行内容修改（文档是记录，改代码才是修复）
- 在修改关键节点（如重连触发、重连成功、失败异常等）添加清晰、可追溯的日志输出

## 工作流

```
接收用户 Bug 描述
    ↓
步骤 1：定位——复现 Bug 并定位根因
    ↓
步骤 2：分析——分析影响范围与修复方案
    ↓
步骤 3：修复——实施代码修改
    ↓
步骤 4：验证——确认修复有效且不引入回归
    ↓
输出修复记录文档 + 变更文件清单
```

### 步骤说明

#### 步骤 1：定位（Locate）
- 确认 Bug 现象，必要时要求用户提供截图、日志、操作步骤
- 通过代码阅读、日志分析、数据库查询等手段复现 Bug
- **定位问题根因**，找到对应的代码文件及行号
- 记录：问题现象、复现步骤、根因位置（文件 + 行号）

#### 步骤 2：分析（Analyze）
- 分析 Bug 影响范围（是否影响其他功能、是否涉及全局性问题）
- 判断是否需要系统性修复（局部修补 vs 全局改造）
- 设计修复方案，评估方案风险
- **向用户说明问题、根因分析和修复方案，获得确认后再推进到步骤 3**

#### 步骤 3：修复（Fix）
- 按修复方案实施代码修改
- 遵循对应层的编码规范（参考前置规范中的规则文件）
- 在关键节点添加必要的日志输出
- 确保新增/修改的代码包含版权注释
- 记录：修改的文件列表及修改内容摘要

#### 步骤 4：验证（Verify）
- 编译项目确认无编译错误
- 执行相关测试用例确认修复有效
- 回归验证：确认不影响相关功能
- 记录：验证结果

## 修复记录文档模板

```markdown
# {Bug 简述}

## 1. 问题描述

{用户反馈的 Bug 现象、截图、日志等}

## 2. 根因分析

| 项目 | 内容 |
|------|------|
| 根因文件 | `{文件路径:行号}` |
| 根因类型 | 空指针 / 逻辑错误 / 并发问题 / SQL 异常 / 前端逻辑 / 其他 |
| 根因说明 | {详细分析根因产生的原因} |

## 3. 影响范围

{评估该 Bug 影响的模块、功能、用户范围}

## 4. 修复方案

{描述修复思路和具体方案}

## 5. 变更文件清单

| 文件 | 操作 | 修改内容摘要 |
|------|------|-------------|
| `{文件路径}` | 新增/修改/删除 | {修改内容简述} |

## 6. 验证结果

- [ ] 编译通过
- [ ] 相关测试通过
- [ ] 回归验证通过

{验证过程和结果描述}
```

## 输出文件

| 文件 | 路径 |
|------|------|
| 修复记录文档 | `Documents/需求分析/BugFix/{yyyyMMdd-HHmmss}-bugfix-{简述}.md` |
| 变更文件 | （步骤 5 实时输出到修复记录文档的变更文件清单表格中） |

## 约束

- 文档文件名必须使用 `{yyyyMMdd-HHmmss}-bugfix-{简述}.md` 格式
- 时间戳必须通过 PowerShell 命令 `powershell -Command "Get-Date -Format 'yyyyMMdd-HHmmss'"` 获取
- 存储路径必须为 `Documents/需求分析/BugFix/`
- 文档必须包含版权注释（第一行：`# Copyright (c) 2020-06-29 Qoder. All rights reserved.`）
- **步骤 2 完成并获用户确认前，禁止直接执行步骤 3 的修改操作**
- 修复代码必须遵循对应层的编码规范（参考前置规范中的规则文件）
- 变更文件清单必须包含所有修改的文件，不得遗漏
- 若 Bug 涉及数据库数据查询，可使用 `query-db` skill 辅助排查
- 每个修改步骤完成后，建议同步执行相关测试以验证修改正确性
