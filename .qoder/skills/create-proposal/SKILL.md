# create-proposal

## 目的

根据用户输入的需求描述，自动生成结构化的需求设计文档，规范需求分析流程，确保文档结构完整、内容可追溯。

## 前置规范

**必须先读取并严格遵循**以下规则文件：

1. `.qoder/rules/md-naming-format.mdc` — MD 文档命名规范（yyyyMMdd-HHmmss-{name}.md）
2. `.qoder/rules/api-restful.mdc` — RESTful API 设计规范（接口设计部分参考）
3. `.qoder/rules/java-early-return.mdc` — Java 代码规范（约束技术方案中的代码风格）
4. `.qoder/rules/java-api-logging.mdc` — API 层日志规范（约束技术方案中的 API 实现）

## 工作流

```
接收用户需求描述
    ↓
步骤 1：澄清需求范围与核心术语
    ↓
步骤 2：分析涉及模块（后端/前端/数据库）
    ↓
步骤 3：生成需求设计文档
    ↓
步骤 4：输出到 Documents/需求分析/需求/
```

### 步骤说明

1. **澄清需求范围与核心术语**
   - 与用户确认需求背景、目标用户、核心业务概念
   - 明确需求边界（做什么/不做什么）
   - 记录关键术语表

2. **分析涉及模块**
   - 后端：涉及的模块（api / business / domain / component-*）
   - 前端：涉及的页面、路由、组件
   - 数据库：涉及的表和字段变更

3. **生成需求设计文档**
   - 文档包含以下固定章节：

4. **输出到 Documents/需求分析/需求/**
   - 文件命名：`{yyyyMMdd-HHmmss}-{需求名}.md`
   - 存储路径：`Documents/需求分析/需求/`

## 文档模板

```markdown
# {需求名称}

## 1. 背景与目标

{描述需求背景、业务痛点、预期目标}

## 2. 功能清单

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|--------|--------|------|
| 模块 A | 功能 1 | P0/P1/P2 | 简要说明 |
| 模块 B | 功能 2 | P0/P1/P2 | 简要说明 |

## 3. 涉及模块

### 后端

| 模块 | 作用 | 变更内容 |
|------|------|---------|
| api | Controller 层 | 新增/修改接口 |
| business | Service 层 | 新增业务逻辑 |
| domain | Entity/Mapper | 新增实体/映射 |
| component-* | 组件层 | 新增组件服务 |

### 前端

| 页面 | 路由 | 说明 |
|------|------|------|
| {页面名称} | /{path} | 页面功能说明 |

### 数据库

| 表名 | 操作 | 说明 |
|------|------|------|
| {table_name} | CREATE/ALTER | 字段说明 |

## 4. 技术方案

{描述整体技术方案，含关键代码示例或架构图}

## 5. 数据库变更

{列出具体 SQL 变更，包含字段名、类型、注释}

## 6. 接口设计

### {接口名称}

```
GET/POST/PUT/DELETE /api/{resources}
```

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| {field} | {type} | 是/否 | 说明 |

**响应示例：**

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```
```

## 输出文件

| 文件 | 路径 |
|------|------|
| 需求文档 | `Documents/需求分析/需求/{yyyyMMdd-HHmmss}-{需求名}.md` |

## 约束

- 文档文件名必须使用 `yyyyMMdd-HHmmss-{name}.md` 格式
- 时间戳必须通过 PowerShell 命令 `Get-Date -Format 'yyyyMMdd-HHmmss'` 获取
- 存储路径必须为 `Documents/需求分析/需求/`（注意是"需求"目录，不是"方案"或"BugFix"）
- 文档第一行必须包含版权注释 `# Copyright (c) 2020-06-29 Qoder. All rights reserved.`
- 文档必须包含完整的 6 个章节：背景、功能清单、涉及模块、技术方案、数据库变更、接口设计
- 接口设计必须遵循 RESTful 规范（参考 api-restful.mdc）
- 生成前必须与用户确认需求理解无误后再生成文档
